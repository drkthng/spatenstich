- Create a minimal `app/src/stores/captureStore.ts` (Zustand):
  ```typescript
  import { create } from 'zustand';
  interface CaptureState {
    photos: string[];   // resized local URIs
    addPhoto: (uri: string) => void;
    removePhoto: (index: number) => void;
    reset: () => void;
  }
  export const useCaptureStore = create<CaptureState>((set) => ({
    photos: [],
    addPhoto: (uri) => set((s) => ({ photos: [...s.photos, uri] })),
    removePhoto: (index) => set((s) => ({ photos: s.photos.filter((_, i) => i !== index) })),
    reset: () => set({ photos: [] }),
  }));
  ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Device camera/gallery -> App | Photos from device enter app; resized and EXIF-stripped before storage |
| App -> Supabase (budget query) | Client-side budget check is UX-only, not authoritative |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-4-03-01 | Information Disclosure | EXIF/GPS in captured photos | mitigate | resizeToMaxMp runs BEFORE enqueuePhoto; enqueuePhoto calls stripExifAndExtractGps (existing Phase 3 implementation). GPS only extracted if optIn=true (D-24/NFR-05). |
| T-4-03-02 | Denial of Service | Client bypasses budget check | accept | Client-side budget check is UX convenience only. Authoritative check is in Edge Function (Plan 02). Client cannot bypass server-side limit. |
| T-4-03-03 | Tampering | User submits dimensions > 200m | mitigate | Client-side validation (> 0, <= 200). Server-side: no constraint needed (non-security, just data quality). |
| T-4-03-04 | Spoofing | Unauthenticated access to capture flow | mitigate | gardenPlanRepo.saveDimensions calls assertAccount(mode). enqueuePhoto checks userId. Expo Router guards in _layout.tsx redirect unauthenticated users. |
</threat_model>
