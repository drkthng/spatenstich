## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client -> Edge Function (via pgmq) | Job payload crosses trust boundary; Edge Function validates |
| Edge Function -> Anthropic API | Photos sent to external AI service (transient, deleted after) |
| Edge Function -> Supabase Storage | Downloads photos with SERVICE_ROLE (bypasses RLS) — path from ai_jobs.payload (server-controlled) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-4-02-01 | Tampering | ai_jobs.payload manipulation | mitigate | Payload set by server-side RPC enqueue_photo_analysis (SECURITY DEFINER), not by client directly. Storage path validated by download attempt. |
| T-4-02-02 | Information Disclosure | CLAUDE_API_KEY in logs | mitigate | T-2-03-04 pattern: never log key. Error messages use err.message only. |
| T-4-02-03 | Information Disclosure | Photos sent to Anthropic | mitigate | Files deleted in finally block (Pitfall 6). Anthropic Files API is transient. EXIF already stripped client-side. |
| T-4-02-04 | Denial of Service | Budget exhaustion | mitigate | Hard limit 200/day per garden_id enforced BEFORE Claude call. Job marked 'failed' with 'daily_limit_exceeded'. |
| T-4-02-05 | Elevation of Privilege | Crafted payload to read other garden's photos | mitigate | Storage path comes from ai_jobs.payload which is set by SECURITY DEFINER RPC with member-check. SERVICE_ROLE download is path-isolated by garden_id prefix. |
| T-4-02-06 | Tampering | Prompt injection via photo metadata | accept | EXIF stripped client-side (Phase 3). Photo content itself could theoretically influence Claude, but output is JSON-parsed and validated by parseElements — malformed output is discarded. Risk is low (garbled elements, not code execution). |
