# Phase 2.5: Shared Garden Model — Research

**Researched:** 2026-04-23
**Supabase Project:** `vitrqkzxkiqvadqfzrcx` (Frankfurt, EU)
**Domain:** Postgres-Schema-Migration + RLS-Refactor + Invite-Flow für 2-User Shared Garden
**Confidence:** HIGH (alle Kernempfehlungen über Supabase-Docs oder offizielle Postgres-Docs verifiziert)

## Summary

Phase 2.5 ist keine Feature-Phase, sondern ein **Datenmodell-Umbau unterhalb laufender Phase-02-Screens**. Das Risiko liegt fast komplett in der einen Migration 003, die in einer einzigen Postgres-Transaktion (1) neue Tabellen anlegt, (2) Default-Gärten für Bestandsuser seedet, (3) `garden_id` auf Phase-2-Tabellen hinzufügt + backfilled + NOT-NULL-t, (4) alle alten `auth.uid() = user_id`-RLS-Policies droppt und Member-Check-Policies anlegt. Dazwischen darf kein Zeitfenster mit kaputter RLS existieren.

Die Recherche bestätigt: Supabase-Migrations laufen als implizite Transaktion (psql `-v ON_ERROR_STOP=1 --single-transaction`-Äquivalent); DDL und DML dürfen im selben SQL-File stehen. Das 2-Member-Limit wird über `BEFORE INSERT TRIGGER` umgesetzt (CHECK mit Subquery ist in Postgres verboten). Der Invite-Code-Consume läuft als `UPDATE … WHERE consumed_at IS NULL RETURNING garden_id` — atomar ohne `SELECT FOR UPDATE`. `auth.uid()` funktioniert in BEFORE-UPDATE-Triggern wenn der Aufruf über `supabase-js` kommt (JWT-Claims sind im Request-Context), returnt aber `null` bei direkten DB-Queries. RLS-Policies werden mit `(select auth.uid())`-Wrap gebaut, damit der Query-Planner initPlan-Caching nutzen kann.

**Primary recommendation:** Eine einzige Migration `20260423000003_shared_garden.sql` mit den DDL+DML-Schritten in der dokumentierten Reihenfolge; Postgres-Function `consume_invite_code(text)` als SECURITY DEFINER (kein Edge Function Deploy nötig); Invite-Code-Generator in derselben Function als `gen_invite_code(text)` mit Crockford-Base32-Alphabet; Migration mit **zwei existierenden Test-Usern** in Supabase-DB (aus `rls_phase2.sql`) ausprobieren, bevor auf Production angewendet wird.

## User Constraints (from CONTEXT.md)

### Locked Decisions

**Garten-Modell & Profil-Scope**

- **D-01:** PLZ + Klimazone + Archetyp wandern komplett auf die neue `gardens`-Tabelle. `profiles` behält nur `id` + `display_name` (neu) + `created_at`/`updated_at`. Semantik: ein Kleingarten hat genau eine Lage/Klimazone/Charakter — nicht pro Person.
- **D-02:** Die Phase-2-Tabellen `vereinsregeln`, `ai_jobs`, `ai_results` bekommen einen `garden_id uuid not null references gardens(id) on delete cascade` — RLS wird von `auth.uid() = user_id` auf `auth.uid() in (select user_id from garden_members where garden_id = <row>.garden_id)` umgestellt.
- **D-03:** `gardens`-Tabelle minimal: `id uuid pk`, `name text not null`, `created_at timestamptz`, `updated_at timestamptz`, `created_by_user_id uuid references auth.users(id)` (Owner-Marker). Keine Parzellen-Metadaten (Maße/Form) — das kommt in Phase 4.
- **D-04:** `garden_members`-Tabelle: `garden_id uuid fk`, `user_id uuid fk`, `role text check (role in ('owner','member'))`, `joined_at timestamptz`. Composite PK `(garden_id, user_id)`. DB CHECK constraint oder Trigger, der Member-Count auf 2 limitiert.
- **D-05:** Neue Spalte `display_name text` in `profiles`, User-gesetzt beim Onboarding (oder migriert aus E-Mail-Prefix als Default). Wird für "zuletzt bearbeitet von"-Label genutzt.
- **D-06 (Claude's Discretion):** FK-Struktur auf geteilten Tabellen — `garden_id` (für RLS-Scope) + `updated_by_user_id` (für SC4-Audit, nullable für Legacy-Zeilen). Bestehendes `user_id` auf `vereinsregeln`/`ai_jobs`/`ai_results` kann als `created_by_user_id` umbenannt bleiben oder droppen — entscheidet Researcher je nach Migration-Komplexität.

**Invite-Code-Flow**

- **D-07:** Nur Owner (entsprechend `garden_members.role = 'owner'`) darf Invite-Code erzeugen.
- **D-08:** 6-stelliger alphanumerischer Code aus ambiguitätsfreiem Alphabet (kein `0/O`, kein `1/I/l`). Am Telefon diktierbar.
- **D-09:** 24h gültig, Ein-Use (DB-Spalte `consumed_at timestamptz null`). Nach Beitritt oder Ablauf nicht wiederverwendbar.
- **D-10:** UI-Flow: **Owner-Seite** Settings → "Mein Garten" → Button "Person einladen" → erzeugt Code, zeigt + Copy/Share. **Joiner-Seite** "Code eingeben"-Screen erreichbar im Onboarding + nachträglich in Settings.
- **D-11:** Revokation implizit: "Neuen Code erzeugen" invalidiert den alten (nur ein aktiver Code pro Garten gleichzeitig). Kein separater "Widerrufen"-Button im MVP.

**Migration Phase 2 → 2.5**

- **D-12:** Auto-Migration für existing Account-User: Default-Garten beim ersten Login nach Deploy (DB-Trigger + App-Boot-Logic, Defense-in-Depth).
- **D-13:** Lokal-Modus bleibt konzeptionell single-user — `gardens`/`garden_members` existieren nur in Supabase. `migrateLocalToAccount.ts` wird erweitert.
- **D-14:** Frau's eigene Daten (falls sie vor Beitritt eigenen Account oder Lokal-Modus hatte) bleiben unangetastet in ihrem eigenen Space. Beim Beitritt via Invite-Code wird ein weiterer `garden_members`-Eintrag für Dirks Garten angelegt. UI zeigt nach Beitritt nur Dirks Garten (aktiver Garten via `authStore.activeGardenId`). Merge zwischen Gärten ist Out-of-Scope.
- **D-15:** RLS-Policy-Switch atomic in EINER Migration.

**Member-Rollen, Limits & LWW-Anzeige**

- **D-16:** Rollen: **Owner** darf Invite-Code erzeugen, Member entfernen, Garten löschen, Owner-Status abgeben. **Member** darf alle Content-Rows bearbeiten + sich selbst aus dem Garten entfernen.
- **D-17:** Max 2 Member pro Garten hart codiert via DB CHECK oder BEFORE INSERT trigger.
- **D-18:** LWW-Tracking: `updated_at timestamptz` + `updated_by_user_id uuid references auth.users(id)` auf allen Tabellen mit `garden_id`. `updated_at` per Trigger (`tg_set_updated_at` aus Phase 1). `updated_by_user_id` client-seitig bei jedem Upsert mitgesendet.
- **D-19:** UI-Anzeige "zuletzt bearbeitet von <Name>": inline pro Row, kleingedruckt.

## Architectural Responsibility Map

| Capability | Primary Tier | Secondary Tier | Rationale |
|------------|-------------|----------------|-----------|
| Shared-Garden-Datenmodell | Database (Postgres) | — | Schema + FK-Integrität + RLS gehören dorthin, wo die Daten liegen. |
| Garten-Scoping per RLS | Database (Postgres) | API (supabase-js) | RLS-Policies sind die einzige verlässliche Autorisierung — Client ist nicht vertrauenswürdig. |
| 2-Member-Limit enforcement | Database (Postgres) | — | Muss in der DB erzwungen werden, sonst racebar. `BEFORE INSERT TRIGGER` ist der Standard-Weg, weil `CHECK` keine Subqueries erlaubt. |
| Invite-Code-Generierung (6-stellig, Crockford) | Database (Postgres RPC via SECURITY DEFINER) | API Client | Eine `rpc('create_invite_for_garden', { gid })`-Function vermeidet einen Edge-Function-Deploy und hält Code-Generator + Owner-Check + 24h-Expiry + Revokation-der-alten-Codes in einer atomaren Transaktion. |
| Invite-Code-Consume | Database (Postgres RPC SECURITY DEFINER) | API Client | Atomar via `UPDATE … RETURNING`; Aufrufer darf ohne Garten-Zugehörigkeit aufrufen. |
| Default-Garten-Auto-Migration für Bestandsuser | Database (Migration 003 seed-Schritt) | App (`_layout.tsx` Fallback bei null `activeGardenId`) | Migration stempelt zum Zeitpunkt des Deploys alle Bestandsuser; App-Fallback deckt User ab, die nach Migration neu signUp-en. |
| `updated_by_user_id`-Befüllung | App (client-seitig im Upsert-Payload) | Database (BEFORE UPDATE TRIGGER als Absicherung) | `auth.uid()` im Trigger-Kontext funktioniert nur bei supabase-js-Aufrufen (JWT-Claim); client-side ist verlässlicher für "erstes Signal", Trigger als Fallback für Consistency. |
| `updated_at`-Pflege | Database (existing `tg_set_updated_at`) | — | Existiert aus Phase 1, wird nur erweitert um die neuen Tabellen. |
| `activeGardenId`-State | App (Zustand / authStore-Erweiterung) | AsyncStorage persist | Client-seitiger Single-Garden-Pointer; Default = erster Garten aus `garden_members WHERE user_id = auth.uid()`. |
| UI "zuletzt bearbeitet von" | App (NativeWind-Komponente, Phase 2 Pattern) | — | Pures Rendering von DB-Feldern; kein Backend-Arbeit. |

**Warum das wichtig ist:** Die Verlockung, Invite-Code-Logik in einer Edge Function zu bauen, ist real (analog zu `extract-vereinsregeln`). Für Phase 2.5 ist sie aber **falsch**: die Funktion ist pure Datenlogik mit Autorisierungsprüfung — genau wofür Postgres-Functions mit `SECURITY DEFINER` existieren. Ein Edge-Function-Deploy würde die Migration-Atomicity brechen (Edge Functions sind separat deploybar, nicht Teil der SQL-Transaktion) und bräuchte einen HTTP-Round-Trip statt einem PostgREST-RPC-Call. Entscheidung unten in §3.

### Alternatives Considered

| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| Postgres RPC für Invite-Code-Generator | Deno Edge Function `create-invite` | Edge Function ist für I/O-heavy/external-API-Calls optimiert; hier ist die Arbeit rein Datenlogik. Edge Function = mehr Deploy-Schritt, mehr Cold-Start-Latenz, keine Transaktions-Integration mit Migration. RPC = ein SQL-File, PostgREST-exposed automatisch, volltransaktional. [CITED: https://www.closefuture.io/blogs/supabase-database-vs-edge-functions — "Use DB Functions for data logic; Edge Functions for external APIs"] |
| BEFORE INSERT Trigger für 2-Member-Limit | Partial unique index | Ein Partial Unique Index kann "nur ein Owner pro Garten" enforcen, aber nicht "maximal 2 Member". Dafür braucht man eine aggregate-Count-Check, und das kann nur ein Trigger. [CITED: Postgres Docs — CHECK constraints können keine Subqueries] |
| Client-side `updated_by_user_id` + optionaler Trigger | Pure Trigger mit `auth.uid()` | Pure Trigger ist eleganter, aber `auth.uid()` ist im Trigger-Kontext unzuverlässig (returnt null bei direkten DB-Queries, z.B. Admin-Scripts / Tests). Client-side Pflicht mit Trigger-Fallback ist Defense-in-Depth. [CITED: supabase/discussions/9066, #22769] |

### Pattern 1: Migration-Atomicity (D-15) — das kritischste Pattern der Phase

**What:** Alle Schema- + Daten- + RLS-Änderungen in EINER `.sql`-Datei, eine implizite Postgres-Transaktion. Wenn irgendein Schritt fehlschlägt, wird rolled back und die DB bleibt auf Phase-2-Stand.

**When to use:** Immer wenn ein RLS-Refactor mit Bestandsdaten-Backfill kombiniert wird.

**Fact-Check:** Supabase führt Migrationen per Default als Transaktion aus — `supabase migration up` wraps jedes File in `BEGIN; ... COMMIT;` (analog zu psql `--single-transaction`). DDL-Befehle wie `CREATE TABLE` / `ALTER TABLE` / `CREATE POLICY` / `DROP POLICY` sind in Postgres transaktional (anders als MySQL). Das bedeutet: wir können im selben File `CREATE TABLE`, `INSERT`, `ALTER TABLE`, `UPDATE`, `ALTER TABLE SET NOT NULL`, `DROP POLICY`, `CREATE POLICY` nacheinander schreiben — entweder alles gelingt, oder nichts wird angewendet. [CITED: https://www.postgresql.org/docs/current/sql-altertable.html — "ALTER TABLE changes the definition of an existing table. Several subforms are described below. Note that the lock level required may differ for each subform."; verifiziert im Supabase-Migration-Doc: https://supabase.com/docs/guides/deployment/database-migrations]

**Reihenfolge in `20260423000003_shared_garden.sql`:**

```sql
-- Schritt 1: Neue Tabellen erstellen
CREATE TABLE public.gardens (...);
CREATE TABLE public.garden_members (...);
CREATE TABLE public.invite_codes (...);

-- Schritt 2: 2-Member-Limit-Trigger anlegen
CREATE FUNCTION public.tg_garden_members_limit_2() RETURNS TRIGGER ...;
CREATE TRIGGER ... BEFORE INSERT ON public.garden_members ...;

-- Schritt 3: RLS auf neuen Tabellen
ALTER TABLE public.gardens ENABLE ROW LEVEL SECURITY;
CREATE POLICY "gardens_member_select" ON public.gardens ...;
-- (siehe §6 Code Examples)

-- Schritt 4: profiles refactor — neue Spalte display_name
ALTER TABLE public.profiles ADD COLUMN display_name text;
-- NB: plz/klimazone/archetype werden bewusst NOCH NICHT gedroppt — siehe "Zwei-Phase-Refactor" unten

-- Schritt 5: Default-Gärten + garden_members-Rows für alle Bestandsuser seeden
INSERT INTO public.gardens (id, name, created_by_user_id)
SELECT gen_random_uuid(), 'Mein Garten', id FROM public.profiles;

INSERT INTO public.garden_members (garden_id, user_id, role)
SELECT g.id, g.created_by_user_id, 'owner'
FROM public.gardens g;

-- Schritt 6: garden_id + updated_by_user_id auf Ziel-Tabellen
ALTER TABLE public.vereinsregeln ADD COLUMN garden_id uuid REFERENCES public.gardens(id) ON DELETE CASCADE;
ALTER TABLE public.vereinsregeln ADD COLUMN updated_by_user_id uuid REFERENCES auth.users(id);
ALTER TABLE public.vereinsregeln ADD COLUMN updated_at timestamptz NOT NULL DEFAULT now();

UPDATE public.vereinsregeln v
SET garden_id = (
  SELECT g.id FROM public.gardens g WHERE g.created_by_user_id = v.user_id LIMIT 1
);

ALTER TABLE public.vereinsregeln ALTER COLUMN garden_id SET NOT NULL;

-- (gleiche Sequenz für ai_jobs, ai_results, profiles.garden_id via profiles-pivoted-approach)

-- Schritt 7: Alte Policies droppen + neue Member-Check-Policies anlegen
DROP POLICY IF EXISTS "vereinsregeln_own" ON public.vereinsregeln;
CREATE POLICY "vereinsregeln_member" ON public.vereinsregeln FOR ALL USING (
  (select auth.uid()) IN (select user_id from public.garden_members where garden_id = vereinsregeln.garden_id)
) WITH CHECK (
  (select auth.uid()) IN (select user_id from public.garden_members where garden_id = vereinsregeln.garden_id)
);
-- (idem für ai_jobs, ai_results, gardens, garden_members)

-- Schritt 8: RPC-Functions
CREATE OR REPLACE FUNCTION public.create_invite_for_garden(gid uuid) ...;
CREATE OR REPLACE FUNCTION public.consume_invite_code(c text) ...;
CREATE OR REPLACE FUNCTION public.ensure_default_garden_for_user() ...;

-- Schritt 9: updated_at Trigger auf neue Tabellen + Bestands-Tabellen
CREATE TRIGGER gardens_updated_at BEFORE UPDATE ON public.gardens
  FOR EACH ROW EXECUTE FUNCTION public.tg_set_updated_at();
CREATE TRIGGER vereinsregeln_updated_at BEFORE UPDATE ON public.vereinsregeln
  FOR EACH ROW EXECUTE FUNCTION public.tg_set_updated_at();
-- (idem ai_jobs, ai_results — falls noch nicht vorhanden)

-- Schritt 10 (OPTIONAL): alte user_id-Spalten auf vereinsregeln/ai_jobs/ai_results
-- KEEP (D-06 Empfehlung): als created_by_user_id umbenennen für Audit. NICHT droppen.
ALTER TABLE public.vereinsregeln RENAME COLUMN user_id TO created_by_user_id;
-- NB: existing Indexe auf user_id werden automatisch mit umbenannt.
```

**Zwei-Phase-Refactor für `profiles`:** PLZ/Klimazone/Archetyp liegen aktuell auf `profiles` (Phase 2) und sollen auf `gardens` wandern (D-01). Der sichere Pfad ist:

1. **In Migration 003:** `profiles` behält PLZ/Klimazone/Archetyp **kopiert** rüber auf `gardens` — die Spalten bleiben auf `profiles` unverändert (Bestands-App funktioniert weiter).
2. **In Phase 3 oder einer Folge-Migration 004:** sobald der App-Code komplett auf `gardens`-Scope umgestellt ist, werden `profiles.plz/klimazone/archetype` gedroppt.

Warum: falls der App-Code nach dem Migration-Deploy zurückrollt (Hotfix auf Phase-2-Version), sind die Spalten noch da und alte Builds funktionieren. Risk-Minimierung wichtiger als Clean-Schema im ersten Shot.

### Pattern 2: 2-Member-Limit via BEFORE INSERT Trigger (D-17)

**What:** Ein `BEFORE INSERT TRIGGER` auf `garden_members`, der vor jeder neuen Row prüft, ob schon 2 Member existieren, und falls ja, einen Error wirft.

**When to use:** Wenn ein Aggregate-Condition (COUNT, SUM) enforced werden muss, das nicht in einer `CHECK`-Constraint-Subquery erlaubt ist.

**Fact-Check:** [CITED: Postgres Docs — "The search condition [of a CHECK constraint] should involve only the columns of the inserted or updated row, … references to other tables are not allowed"]. Partial Unique Index funktioniert nur für "höchstens EIN mit Bedingung X" (hier: höchstens ein Owner), nicht für "höchstens ZWEI Member insgesamt".

**Race-Condition-Verhalten:** Der Trigger läuft innerhalb der INSERT-Transaktion. Zwei parallele INSERTs auf denselben `garden_id` könnten theoretisch beide COUNT=1 sehen und beide einfügen → Ergebnis COUNT=3. Mitigation: **Advisory Lock** am Trigger-Start, oder **SERIALIZABLE**-Isolation, oder (pragmatisch) **`SELECT … FOR UPDATE` auf der gardens-Row** innerhalb des Triggers, damit parallele Inserts serialisieren. Für 2-User-MVP mit nur einem Owner-User, der Invites erzeugt, ist das Risiko praktisch nicht existent — aber wir bauen die Lock-Variante trotzdem, weil es kostenlos ist.

```sql
CREATE OR REPLACE FUNCTION public.tg_garden_members_limit_2()
RETURNS trigger LANGUAGE plpgsql AS $$
DECLARE
  current_count int;
BEGIN
  -- Row-Lock auf gardens serialisiert parallele Inserts auf denselben garden_id.
  PERFORM 1 FROM public.gardens WHERE id = NEW.garden_id FOR UPDATE;

  SELECT count(*) INTO current_count
  FROM public.garden_members
  WHERE garden_id = NEW.garden_id;

  IF current_count >= 2 THEN
    RAISE EXCEPTION 'garden_member_limit_exceeded: garden % already has 2 members', NEW.garden_id
      USING ERRCODE = '23514'; -- check_violation (SQLSTATE), so supabase-js surfaces a clean error
  END IF;

  RETURN NEW;
END $$;

CREATE TRIGGER garden_members_limit_2
  BEFORE INSERT ON public.garden_members
  FOR EACH ROW EXECUTE FUNCTION public.tg_garden_members_limit_2();
```

**Error-Surface für Client:** `supabase-js` returnt bei `23514` einen `PostgrestError` mit `code = '23514'` und `message` = unserer RAISE-Message. UI kann darauf matchen und "Garten ist voll" zeigen.

### Pattern 3: Invite-Code als Postgres-RPC mit SECURITY DEFINER

**What:** Zwei PL/pgSQL-Functions `create_invite_for_garden` (Owner-Only) und `consume_invite_code` (von jedem authentifizierten User aufrufbar). SECURITY DEFINER, damit sie RLS selbst enforcen können ohne dass der Client direkt auf `invite_codes` schreiben muss.

**When to use:** Wenn Autorisierung komplex genug ist, dass sie nicht sauber als RLS-Policy ausdrückbar ist — hier: "Owner darf generieren und nur für seinen Garten", "Konsumiert wird von einem Noch-Nicht-Member".

**Fact-Check:**
- RPCs werden automatisch via PostgREST exponiert — kein Deploy-Schritt nötig. [CITED: https://supabase.com/docs/guides/database/functions]
- SECURITY DEFINER läuft mit den Privilegien des Function-Owners (postgres-superuser bei Supabase), was RLS bypassed. Best-Practice ist: `SET search_path = public, pg_temp` in der Function setzen, um SQL-Injection via search_path-Manipulation zu verhindern. [CITED: https://supabase.com/docs/guides/database/functions — "Always set the search_path"]
- `auth.uid()` ist in der Function verfügbar, wenn der Aufruf über `supabase-js.rpc()` erfolgt (JWT wird an PostgREST weitergereicht). [VERIFIED: supabase/discussions/22769 + supabase/discussions/9066]

**Code-Generator mit Crockford-Base32-Alphabet:**

Crockford-Base32 Alphabet: `0123456789ABCDEFGHJKMNPQRSTVWXYZ` (32 Zeichen, keine I/L/O/U). Für unsere Anforderung "keine 0" → entferne zusätzlich `0`, ergibt `123456789ABCDEFGHJKMNPQRSTVWXYZ` (31 Zeichen). 6 Zeichen aus 31 = 31^6 ≈ 887 Mio Kombinationen — genug Entropie für ein 24h-gültiges Ein-Use-Token.

```sql
CREATE OR REPLACE FUNCTION public.gen_invite_code() RETURNS text
LANGUAGE plpgsql AS $$
DECLARE
  alphabet text := '123456789ABCDEFGHJKMNPQRSTVWXYZ'; -- kein 0/O/I/L/U
  result text := '';
  rnd_bytes bytea;
  idx int;
  i int;
BEGIN
  rnd_bytes := gen_random_bytes(6);  -- pgcrypto; 6 Bytes = 6 Zeichen
  FOR i IN 0..5 LOOP
    idx := (get_byte(rnd_bytes, i) % length(alphabet)) + 1;
    result := result || substring(alphabet from idx for 1);
  END LOOP;
  RETURN result;
END $$;
```

Hinweis: `gen_random_bytes(6)` liefert kryptographisch starke Zufallsbytes (pgcrypto). Der `%`-Bias ist bei 256 % 31 marginal ungleich verteilt; für eine Anti-Guessing-Sicherheit ist das vernachlässigbar (Angreifer bräuchte ~1 Mio Versuche um 50%-Trefferchance zu haben, und wir rate-limiten nicht — aber der Code ist 24h gültig und Ein-Use). Akzeptabel für MVP.

### Pattern 4: Atomarer Consume ohne SELECT FOR UPDATE (D-09)

**What:** Invite-Code-Consume ist ein einziges `UPDATE … SET consumed_at = now() WHERE code = $1 AND consumed_at IS NULL AND expires_at > now() RETURNING garden_id`. Kein vorheriges SELECT, kein explizites Lock.

**When to use:** Single-Use-Token-Consume. Pattern ist race-safe ohne expliziten SELECT FOR UPDATE.

**Fact-Check:** [CITED: https://www.cybertec-postgresql.com/en/transaction-anomalies-with-select-for-update/] Unter READ COMMITTED (Postgres-Default): "a single row update acquires a database-wide write lock and performs the update only if the WHERE expression is true, so this is atomic in all isolation levels. For each row it finds, it obtains a write lock. After it obtains the write lock, it re-evaluates the condition on the most recent version of this row. If it still matches, the update is applied."

Das heißt: wenn zwei Clients parallel denselben Code konsumieren wollen, gewinnt genau einer (der andere sieht nach Re-Evaluation `consumed_at IS NOT NULL` und matched nicht mehr). Der zweite Client bekommt `RETURNING garden_id` mit 0 Rows zurück → Error "code_already_consumed_or_expired".

```sql
CREATE OR REPLACE FUNCTION public.consume_invite_code(p_code text)
RETURNS uuid  -- returns garden_id
LANGUAGE plpgsql SECURITY DEFINER SET search_path = public, pg_temp AS $$
DECLARE
  v_garden_id uuid;
  v_user_id uuid;
BEGIN
  v_user_id := auth.uid();
  IF v_user_id IS NULL THEN
    RAISE EXCEPTION 'not_authenticated' USING ERRCODE = '42501'; -- insufficient_privilege
  END IF;

  UPDATE public.invite_codes
  SET consumed_at = now(), consumed_by_user_id = v_user_id
  WHERE upper(code) = upper(p_code)
    AND consumed_at IS NULL
    AND expires_at > now()
  RETURNING garden_id INTO v_garden_id;

  IF v_garden_id IS NULL THEN
    RAISE EXCEPTION 'invite_invalid_or_expired' USING ERRCODE = 'P0002'; -- no_data_found
  END IF;

  -- Member-Insert; BEFORE INSERT Trigger enforced 2-Member-Limit.
  INSERT INTO public.garden_members (garden_id, user_id, role)
  VALUES (v_garden_id, v_user_id, 'member')
  ON CONFLICT (garden_id, user_id) DO NOTHING;  -- idempotent wenn irgendwie re-called

  RETURN v_garden_id;
END $$;

REVOKE ALL ON FUNCTION public.consume_invite_code(text) FROM PUBLIC;
GRANT EXECUTE ON FUNCTION public.consume_invite_code(text) TO authenticated;
```

### Pattern 5: RLS-Policy-Performance — `(select auth.uid())` Wrap

**What:** Jede RLS-Policy, die `auth.uid()` oder `is_admin()` aufruft, wird in ein `(select …)` gewrappt, damit Postgres' Query-Planner das Ergebnis cacht statt pro Row neu zu berechnen.

**When to use:** Immer. Supabase' Performance-Advisor-Lint `0003_auth_rls_initplan` flagged unoptimierte Policies automatisch.

**Fact-Check:** [CITED: https://supabase.com/docs/guides/database/database-advisors?lint=0003_auth_rls_initplan] — "Wrapping the function in SELECT causes an initPlan to be run by the optimizer which allows it to 'cache' the results versus calling the function on each row. … For RLS like `auth.uid() = user_id`, over 100x improvement on large tables."

**Anti-Pattern:**

```sql
-- LANGSAM — auth.uid() wird pro Row aufgerufen
CREATE POLICY "foo" ON public.vereinsregeln FOR ALL
  USING (auth.uid() IN (SELECT user_id FROM public.garden_members WHERE garden_id = vereinsregeln.garden_id));
```

**Pattern:**

```sql
-- SCHNELL — auth.uid() einmal aufgerufen via initPlan
CREATE POLICY "vereinsregeln_member" ON public.vereinsregeln FOR ALL
  USING (
    (select auth.uid()) IN (
      SELECT user_id FROM public.garden_members WHERE garden_id = vereinsregeln.garden_id
    )
  )
  WITH CHECK (
    (select auth.uid()) IN (
      SELECT user_id FROM public.garden_members WHERE garden_id = vereinsregeln.garden_id
    )
  );
```

**Index-Empfehlung:**

```sql
CREATE INDEX garden_members_user_garden_idx ON public.garden_members (user_id, garden_id);
-- Falls öfter per garden_id angefragt:
CREATE INDEX garden_members_garden_idx ON public.garden_members (garden_id);
-- Auf gescoped Tables:
CREATE INDEX vereinsregeln_garden_idx ON public.vereinsregeln (garden_id);
CREATE INDEX ai_jobs_garden_status_idx ON public.ai_jobs (garden_id, status);
-- (ersetzt den Phase-1-Index ai_jobs_user_status_idx? Behalten für historische Queries, neuer Index ergänzend.)
```

### Pattern 6: `updated_by_user_id` — Client-first mit Trigger-Fallback (D-18)

**What:** App sendet `updated_by_user_id: auth.user().id` in jedem `.upsert()`. Zusätzlich existiert ein BEFORE UPDATE Trigger, der das Feld befüllt falls der Client es vergessen hat.

**When to use:** Wenn Authorship-Tracking für UI-Anzeige + Audit benötigt wird.

**Fact-Check:** [CITED: supabase/discussions/9066 + #22769] — `auth.uid()` in Triggern funktioniert, aber NUR wenn der DB-Change über `supabase-js` (REST API) mit gültigem JWT kam. Direkte DB-Queries (z.B. `supabase db query`, Scripts, Admin-Konsole) haben `auth.uid() = null`.

**Konsequenz:** wir dürfen uns **nicht allein** auf den Trigger verlassen. Die TS-Repos (`vereinsregelnRepo.ts`, `gardenRepo.ts`, etc.) müssen `updated_by_user_id` explizit in `toRow()` reinpflegen.

```sql
-- Fallback-Trigger (falls Client es vergessen hat ODER JWT vorhanden ist)
CREATE OR REPLACE FUNCTION public.tg_set_updated_by_user_id()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
  IF NEW.updated_by_user_id IS NULL AND auth.uid() IS NOT NULL THEN
    NEW.updated_by_user_id := auth.uid();
  END IF;
  RETURN NEW;
END $$;

-- Auf jeder Tabelle mit updated_by_user_id:
CREATE TRIGGER vereinsregeln_set_updated_by BEFORE UPDATE ON public.vereinsregeln
  FOR EACH ROW EXECUTE FUNCTION public.tg_set_updated_by_user_id();
```

**Wichtig:** Trigger setzt das Feld NUR wenn es NULL ist — überschreibt NICHT, falls der Client bewusst einen anderen Wert gesetzt hat (sollte nie passieren, aber: Safety).

### Pattern 7: Default-Garten-Auto-Migration (D-12) — Defense-in-Depth

**What:** Zwei unabhängige Mechanismen stellen sicher, dass jeder Account-User genau einen Default-Garten hat:

1. **Migration-003-Seed:** seedet Gärten für alle zum Deploy-Zeitpunkt existierenden `profiles`-Rows.
2. **App-Boot-Logic in `_layout.tsx`:** wenn `session?.user` existiert aber `activeGardenId` ist null (oder `garden_members`-Query returnt 0 Rows) → RPC `ensure_default_garden_for_user()` aufrufen, dann `authStore.setActiveGarden(…)`.

Die App-Boot-Variante deckt zwei Edge-Cases ab:
- Neue signUp nach Deploy (Migration-Seed hat diesen User nicht mehr erreicht).
- Irgendein pathologischer Fall, bei dem `garden_members` leer ist (z.B. nach manueller DB-Intervention).

```sql
CREATE OR REPLACE FUNCTION public.ensure_default_garden_for_user()
RETURNS uuid  -- returns the garden_id (existing or newly created)
LANGUAGE plpgsql SECURITY DEFINER SET search_path = public, pg_temp AS $$
DECLARE
  v_user uuid := auth.uid();
  v_garden_id uuid;
BEGIN
  IF v_user IS NULL THEN
    RAISE EXCEPTION 'not_authenticated' USING ERRCODE = '42501';
  END IF;

  -- Wenn User schon Mitglied eines Gartens ist → erstes zurückgeben (D-14, first-garden-wins)
  SELECT garden_id INTO v_garden_id
  FROM public.garden_members
  WHERE user_id = v_user
  ORDER BY joined_at ASC
  LIMIT 1;

  IF v_garden_id IS NOT NULL THEN
    RETURN v_garden_id;
  END IF;

  -- Sonst neuen Default-Garten erstellen
  INSERT INTO public.gardens (name, created_by_user_id)
  VALUES ('Mein Garten', v_user)
  RETURNING id INTO v_garden_id;

  INSERT INTO public.garden_members (garden_id, user_id, role)
  VALUES (v_garden_id, v_user, 'owner');

  RETURN v_garden_id;
END $$;

REVOKE ALL ON FUNCTION public.ensure_default_garden_for_user() FROM PUBLIC;
GRANT EXECUTE ON FUNCTION public.ensure_default_garden_for_user() TO authenticated;
```

**Idempotenz:** die Function returnt `v_garden_id` vorzeitig falls schon Mitgliedschaft existiert → safe bei mehrfachem Aufruf.

**App-Boot-Integration in `_layout.tsx`:**

```ts
// Pseudocode — konkrete Implementierung durch Planner
function GuardedStack() {
  const { identity, isLoading } = useAuth();
  const { activeGardenId, setActiveGarden } = useAuthStore();

  React.useEffect(() => {
    if (identity && !activeGardenId) {
      supabase.rpc('ensure_default_garden_for_user').then(({ data }) => {
        if (data) setActiveGarden(data);
      });
    }
  }, [identity, activeGardenId]);
  // ...
}
```

### Pattern 8: `migrateLocalToAccount` Erweiterung (D-13)

**What:** Der bestehende atomic-tail-Flow wird erweitert um: nach signUp aber VOR den ersten Upserts → RPC `ensure_default_garden_for_user()` aufrufen → zurückgegebene `garden_id` in alle nachfolgenden `toRow()`-Calls reinpatchen.

**Reihenfolge (NEU):**

1. `supabase.auth.signUp(...)` — wie bisher.
2. `supabase.rpc('ensure_default_garden_for_user')` → `newGardenId`. **NEU** — atomar auf Server-Seite. Idempotent.
3. Read local profile + vereinsregeln aus Storage (wie bisher).
4. Upsert `profiles` mit `id: newUserId, display_name: emailPrefix, ...` (und ggf. PLZ/Klima/Archetyp fürs zweiphasige Refactor).
5. Upsert `gardens` mit `id: newGardenId, name: 'Mein Garten', plz, klimazone, archetype` — NEU (seedet Garten-Metadaten).
6. Upsert `vereinsregeln` mit `toRow(r, newUserId, newGardenId)` — **toRow muss um garden_id erweitert werden**.
7. `authStore.setAccountMode(newUserId)` + `setActiveGarden(newGardenId)`.
8. `storage.delete('profile')` + `storage.delete('vereinsregeln')` — atomic tail unverändert.

**Wichtig:** Schritt 5 (gardens-Upsert mit Metadaten) kommt VOR Schritt 6 (vereinsregeln-Upsert mit garden_id-FK), damit der FK-Constraint erfüllt ist. Schritt 2 stellt sicher, dass der garden-Row existiert bevor Schritt 5 ihn updated.

### Anti-Patterns to Avoid

- **`DROP POLICY` ohne direkt folgendes `CREATE POLICY` im selben File:** öffnet ein Zeitfenster, in dem keine Policy auf der Tabelle ist → RLS zeigt allen authentifizierten Usern NICHTS (weil ohne Policy `ENABLE ROW LEVEL SECURITY` default-deny ist). Fix: Policies im selben Migration-File rotieren, nicht in zwei getrennten Deploys.
- **CHECK-Constraint mit Subquery:** Postgres blockiert das syntaktisch. Nutze Trigger stattdessen.
- **`NOT IN (SELECT …)`-RLS-Policy:** Performance-Killer laut Supabase Advisor `0001_auth_policies_not_in` — Postgres macht dafür keinen Index-Scan. Nutze `EXISTS` oder `IN` (nicht negated).
- **Edge Function für reine Datenlogik:** wie in `extract-vereinsregeln` richtig (I/O zu Claude API), aber für Invite-Code-Flow über-engineered. RPC ist einfacher und transaktional mit anderen DB-Ops.
- **`user_id`-Spalte droppen ohne Migration-Zwischenstand:** Apps die während Rollout auf Phase-2-Version zurückrollen crashen. Als `created_by_user_id` renamen ist sicherer (T-2-04-03-Analogie: rollback-safety).
- **`(select auth.uid())` Wrap vergessen:** Policy funktioniert, aber Performance leidet bei Wachstum. Supabase Lint `0003_auth_rls_initplan` catched es — aber besser gleich richtig.

## Common Pitfalls

### Pitfall 1: RLS-Zeitfenster während Migration (RECHERCHE-FRAGE 1)

**What goes wrong:** Wenn die alte `vereinsregeln_own`-Policy gedroppt wird und die neue `vereinsregeln_member`-Policy einen Syntax-Fehler hat oder auf eine noch-nicht-existierende Tabelle referenziert, haben wir einen Moment mit RLS enabled + keine Policy → alle authenticated Users sehen 0 Rows.
**Why it happens:** Supabase-Migration ist zwar transaktional, aber bei Fehler innerhalb der Transaktion wird alles rolled back — und Entwickler denken, die Datenbank wäre jetzt in einem inkonsistenten Zustand. Ist sie nicht; das Rollback greift. Trotzdem Panik-Risiko.
**How to avoid:** (1) Migration auf Supabase-Staging/Local-Branch probieren bevor auf Production. `supabase db reset --linked` auf Entwickler-Branch, Migration durchlaufen lassen, `rls_phase25_shared_garden.sql` gegen Test-User laufen lassen. (2) Migration in genau dieser Reihenfolge strukturieren: CREATE new objects → SEED data → ADD columns → BACKFILL → NOT NULL → DROP old policy + CREATE new policy.
**Warning signs:** Logs zeigen "permission denied for table vereinsregeln" oder "violates row-level security policy" unmittelbar nach Deploy.

### Pitfall 2: `auth.uid()` gibt `null` im Trigger (RECHERCHE-FRAGE 6)

**What goes wrong:** Ein BEFORE UPDATE Trigger, der `NEW.updated_by_user_id := auth.uid()` setzt, setzt stattdessen NULL, wenn das UPDATE von einem Admin-Script oder einer SQL-Konsole kam (kein JWT).
**Why it happens:** `auth.uid()` ist `current_setting('request.jwt.claim.sub')` — ohne Request-Context ist das NULL.
**How to avoid:** Client sendet `updated_by_user_id` aktiv mit (Pattern 6). Trigger überschreibt NICHT, nur fillt falls NULL.
**Warning signs:** In Production-DB: `SELECT count(*) FROM vereinsregeln WHERE updated_by_user_id IS NULL` > 0 obwohl alle UPDATEs über App liefen → Trigger und/oder Client-Code fehlerhaft.

### Pitfall 3: 2-Member-Limit-Race (RECHERCHE-FRAGE 2)

**What goes wrong:** Zwei parallele `consume_invite_code`-Aufrufe auf denselben Garten fügen beide ein drittes Member hinzu.
**Why it happens:** Der BEFORE INSERT Trigger liest `count(*)` vor dem Insert; ohne Row-Lock auf `gardens` können zwei Transaktionen beide `count = 1` sehen und beide inserten → `count = 3`.
**How to avoid:** `PERFORM 1 FROM gardens WHERE id = NEW.garden_id FOR UPDATE` am Trigger-Start (siehe Pattern 2). Serialisiert parallele Inserts.
**Warning signs:** Sehr selten; in 2-User-MVP praktisch nicht zu reproduzieren. Test: Manual SQL-Test mit zwei parallelen psql-Sessions auf denselben Garten.

### Pitfall 4: `profiles.plz/klimazone/archetype` zu früh gedroppt

**What goes wrong:** Migration 003 dropped die Spalten; App-Code liest aber noch von `profiles` (noch nicht auf `gardens` umgestellt) → Runtime-Crash nach Deploy.
**Why it happens:** Schema-Umbau läuft vor App-Code-Umbau. Zwischen Migration-Apply und App-Build-Deploy gibt es ein Zeitfenster, in dem App-Version N-1 auf Schema N läuft.
**How to avoid:** **Zwei-Phase-Refactor (siehe Pattern 1)**: Migration 003 kopiert PLZ/Klima/Archetyp auf `gardens` UND lässt sie auf `profiles` intakt. App wird umgestellt. Eine spätere Migration 004 droppt die alten Spalten. Phase 2.5 macht nur Teil 1 + App-Umbau.
**Warning signs:** Nach Migration-Deploy: App crasht mit "column 'plz' does not exist" oder liefert PLZ=null zurück (weil App das Feld verlassen wurde aber das Profile-Screen das noch liest).

### Pitfall 5: Supabase-Types out-of-date

**What goes wrong:** TypeScript-Errors "Property 'garden_id' does not exist on type 'Vereinsregeln'" obwohl die Spalte in der DB existiert.
**Why it happens:** `packages/shared/src/types/database.ts` ist auto-generiert — nach Migration 003 muss er regeneriert werden.
**How to avoid:** In den Plan explizit als Task: `supabase gen types typescript --project-id vitrqkzxkiqvadqfzrcx > packages/shared/src/types/database.ts` nach Migration-Apply, vor App-Code-Änderungen.
**Warning signs:** tsc schlägt fehl, TS kann keine `.from('gardens')` resolven.

### Pitfall 6: Alte `user_id`-Spalten in Phase-2-Repos verursachen Dangling-Values

**What goes wrong:** `vereinsregelnRepo.toRow` setzt `user_id` — aber die Spalte heißt nun `created_by_user_id`. supabase-js returnt einen Error "column 'user_id' of relation 'vereinsregeln' does not exist".
**Why it happens:** Wir renamen die Spalte (Pattern 1 Schritt 10). App-Code wurde noch nicht angepasst.
**How to avoid:** In einem einzigen Commit (Atomic): (a) Migration applien, (b) `toRow` in vereinsregelnRepo auf `created_by_user_id` umstellen, (c) Types regenerieren, (d) Repos updaten. Test-Suite muss vor Push grün sein.
**Warning signs:** supabase-js errors mit `42703` (undefined_column).

### Pitfall 7: `display_name` Default (D-05 Claude's Discretion)

**What goes wrong:** Dirks Frau sieht "zuletzt bearbeitet von null" weil sie noch keinen `display_name` gesetzt hat.
**Why it happens:** `display_name` ist optional im Schema.
**How to avoid:** Migration 003 seeded `display_name` für Bestandsuser aus `auth.users.email`-Prefix (z.B. `split_part(email, '@', 1)`). UI zeigt `coalesce(display_name, 'Unbekannt')` oder `'—'`. Onboarding (Phase 2.5 Plan) bietet einen Input-Screen wo User den Namen bei Bedarf ändern kann.

### Pitfall 8: Edge Function `extract-vereinsregeln` bricht nach Migration 003

**What goes wrong:** `extract-vereinsregeln` inserted `vereinsregeln` mit `user_id` aber ohne `garden_id` → NOT NULL violation.
**Why it happens:** Edge-Function-Code wurde nicht angepasst.
**How to avoid:** Entweder (1) Edge Function Code anpassen + redeploy in derselben Phase (aber: Phase 9 ist flagged off, Function wird nicht getriggert — Low-Risk), oder (2) `garden_id DEFAULT` setzen via Trigger (gelooked-up aus `user_id` → `garden_members`). Empfehlung: (1) + Task in Phase 2.5 Plan: "extract-vereinsregeln Edge Function Payload um garden_id erweitern". Feature-Flag off macht die Function ungetriggert — wir können den Fix leise mitshippen.

CREATE OR REPLACE FUNCTION public.gen_invite_code() RETURNS text
LANGUAGE plpgsql AS $$
DECLARE
  alphabet text := '123456789ABCDEFGHJKMNPQRSTVWXYZ';
  result text := '';
  rnd_bytes bytea;
  idx int;
  i int;
BEGIN
  rnd_bytes := gen_random_bytes(6);
  FOR i IN 0..5 LOOP
    idx := (get_byte(rnd_bytes, i) % length(alphabet)) + 1;
    result := result || substring(alphabet from idx for 1);
  END LOOP;
  RETURN result;
END $$;

CREATE OR REPLACE FUNCTION public.create_invite_for_garden(p_garden_id uuid)
RETURNS text
LANGUAGE plpgsql SECURITY DEFINER SET search_path = public, pg_temp AS $$
DECLARE
  v_user uuid := auth.uid();
  v_is_owner boolean;
  v_code text;
  v_attempts int := 0;
BEGIN
  IF v_user IS NULL THEN
    RAISE EXCEPTION 'not_authenticated' USING ERRCODE = '42501';
  END IF;

  -- Owner-Check (D-07)
  SELECT exists(
    SELECT 1 FROM public.garden_members
    WHERE garden_id = p_garden_id AND user_id = v_user AND role = 'owner'
  ) INTO v_is_owner;

  IF NOT v_is_owner THEN
    RAISE EXCEPTION 'not_owner' USING ERRCODE = '42501';
  END IF;

  -- D-11: alte aktive Codes dieses Gartens invalidieren
  UPDATE public.invite_codes
  SET consumed_at = now()
  WHERE garden_id = p_garden_id AND consumed_at IS NULL;

  -- Neuen Code generieren (bei Unique-Collision retry, max 10)
  LOOP
    v_attempts := v_attempts + 1;
    v_code := public.gen_invite_code();
    BEGIN
      INSERT INTO public.invite_codes (garden_id, code, created_by_user_id)
      VALUES (p_garden_id, v_code, v_user);
      EXIT;
    EXCEPTION WHEN unique_violation THEN
      IF v_attempts >= 10 THEN
        RAISE EXCEPTION 'code_generation_failed' USING ERRCODE = 'P0001';
      END IF;
    END;
  END LOOP;

  RETURN v_code;
END $$;

REVOKE ALL ON FUNCTION public.create_invite_for_garden(uuid) FROM PUBLIC;
GRANT EXECUTE ON FUNCTION public.create_invite_for_garden(uuid) TO authenticated;

CREATE OR REPLACE FUNCTION public.consume_invite_code(p_code text)
RETURNS uuid
LANGUAGE plpgsql SECURITY DEFINER SET search_path = public, pg_temp AS $$
DECLARE
  v_user uuid := auth.uid();
  v_garden_id uuid;
BEGIN
  IF v_user IS NULL THEN
    RAISE EXCEPTION 'not_authenticated' USING ERRCODE = '42501';
  END IF;

  UPDATE public.invite_codes
  SET consumed_at = now(), consumed_by_user_id = v_user
  WHERE upper(code) = upper(p_code)
    AND consumed_at IS NULL
    AND expires_at > now()
  RETURNING garden_id INTO v_garden_id;

  IF v_garden_id IS NULL THEN
    RAISE EXCEPTION 'invite_invalid_or_expired' USING ERRCODE = 'P0002';
  END IF;

  -- BEFORE INSERT Trigger enforced 2-Member-Limit
  INSERT INTO public.garden_members (garden_id, user_id, role)
  VALUES (v_garden_id, v_user, 'member')
  ON CONFLICT (garden_id, user_id) DO NOTHING;

  RETURN v_garden_id;
END $$;

REVOKE ALL ON FUNCTION public.consume_invite_code(text) FROM PUBLIC;
GRANT EXECUTE ON FUNCTION public.consume_invite_code(text) TO authenticated;

CREATE OR REPLACE FUNCTION public.ensure_default_garden_for_user()
RETURNS uuid
LANGUAGE plpgsql SECURITY DEFINER SET search_path = public, pg_temp AS $$
DECLARE
  v_user uuid := auth.uid();
  v_garden_id uuid;
  v_display_name text;
BEGIN
  IF v_user IS NULL THEN
    RAISE EXCEPTION 'not_authenticated' USING ERRCODE = '42501';
  END IF;

  SELECT garden_id INTO v_garden_id
  FROM public.garden_members WHERE user_id = v_user
  ORDER BY joined_at ASC LIMIT 1;

  IF v_garden_id IS NOT NULL THEN
    RETURN v_garden_id;
  END IF;

  -- Lazy profile upsert, falls User sich neu registriert hat
  SELECT split_part(email, '@', 1) INTO v_display_name FROM auth.users WHERE id = v_user;
  INSERT INTO public.profiles (id, display_name)
  VALUES (v_user, v_display_name)
  ON CONFLICT (id) DO NOTHING;

  INSERT INTO public.gardens (name, created_by_user_id)
  VALUES ('Mein Garten', v_user)
  RETURNING id INTO v_garden_id;

  INSERT INTO public.garden_members (garden_id, user_id, role)
  VALUES (v_garden_id, v_user, 'owner');

  RETURN v_garden_id;
END $$;

REVOKE ALL ON FUNCTION public.ensure_default_garden_for_user() FROM PUBLIC;

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `auth.uid() = user_id` direkte RLS-Policy | `(select auth.uid()) IN (SELECT user_id FROM members …)` mit SELECT-wrap | Supabase Advisor 0003 (stable seit ~2024) | 100x Speedup auf großen Tabellen; Supabase-Lint flagged unwrapped Policies automatisch. [VERIFIED: https://supabase.com/docs/guides/database/database-advisors?lint=0003_auth_rls_initplan] |
| CHECK constraint mit Subquery | BEFORE INSERT Trigger + optional Row-Lock | Postgres-Design — gilt seit immer | Erzwingt die Nutzung von PL/pgSQL statt declarative Constraint. |
| `SELECT FOR UPDATE` + `UPDATE` in 2 Schritten für Token-Consume | Single `UPDATE … WHERE … RETURNING` | Postgres READ COMMITTED-Semantik — immer so | Weniger Code, kein Deadlock-Potenzial. |
| Edge Function für alles | RPC für reine DB-Logik, Edge Function nur für externe APIs | Community-Best-Practice seit 2024 | Weniger Deploy-Overhead, volltransaktional. [CITED: closefuture.io 2026 comparison] |
| Client-seitige moddatetime für `updated_by` | BEFORE UPDATE Trigger + Client-Payload Defense-in-Depth | Supabase Discussions seit 2023 | Works without surprises wenn API-Calls durchs REST-Gateway gehen; der Client-Pfad deckt Edge-Cases (direkter DB-Access, Admin-Scripts). |

**Deprecated/outdated:**
- Direkte RLS-Policies ohne `(select …)` Wrap — Supabase-Advisor lint 0003 flagged diese.
- `CREATE POLICY ... USING (true)` als Catch-All — unsicher, niemals.

## Open Questions

1. **Edge Function `extract-vereinsregeln` Schreibpfad**
   - What we know: `index.ts` called Claude Files API und returnt geparste Regeln als Response-Body.
   - What's unclear: Ob die Function selbst in `vereinsregeln` schreibt oder nur JSON zurückgibt (Client-seitig upsert). Erste Inspektion (file-head) zeigt nur Download-Flow; Write-Pfad nicht gesehen.
   - Recommendation: Planner inspiziert `supabase/functions/extract-vereinsregeln/index.ts` komplett. Wenn Write vorhanden → Task "Edge Function Payload um garden_id erweitern". Wenn kein Write → nichts zu tun (Client-side Upsert nutzt neues `vereinsregelnRepo.toRow` mit garden_id automatisch).

2. **`zuletzt bearbeitet von`-Timestamp-Format (D-19 Discretion)**
   - What we know: Zwei Optionen im Raum — "vor 2 h" (relative) vs "heute 14:22" (absolute).
   - What's unclear: Welches Pattern hat Phase 2 etabliert? `app/src/lib/utils.ts` + `packages/shared/src/i18n/de.json` checken.
   - Recommendation: Planner entscheidet auf Basis von `utils.ts` — falls `formatRelative`/`formatDistance` schon existiert, wiederverwenden. Sonst minimaler Inline-Helper in einem neuen `app/src/components/lastEditedBy.tsx`.

3. **Default-Garten-Name i18n (Claude's Discretion)**
   - What we know: "Mein Garten" vs "Garten von <display_name>".
   - What's unclear: Zeigt die UI den Garten-Namen prominent? Für 2 User in einem einzigen Garten ist der Name fast irrelevant (es gibt nur einen aktiven).
   - Recommendation: "Mein Garten" als DB-Seed-Default. User kann später umbenennen (Settings → Mein Garten → Name-Edit in Phase 2.5 Plan als Task aufnehmen, oder in eine Folge-Phase deferred — Planner entscheidet).

4. **QR-Code-Generierung (Claude's Discretion)**
   - What we know: Nice-to-have für Invite-Code-Display.
   - What's unclear: Gibt es bereits QR-Libraries in Expo SDK 55? (`react-native-qrcode-svg` ist üblich.)
   - Recommendation: **Deferred — NICHT in Phase 2.5**. Text-Code + Copy + Share-Sheet ist ausreichend für MVP. Planner schiebt QR explizit in Deferred.

5. **Owner-Transfer bei Self-Remove (Claude's Discretion D-19-Cascade)**
   - What we know: Wenn Owner sich selbst entfernt und Member bleibt zurück — Ownership transferieren, oder Garten löschen?
   - What's unclear: Was ist UX-akzeptabel?
   - Recommendation: Phase-2.5-MVP-Entscheidung: **Owner kann sich NICHT selbst entfernen wenn Member existiert**. UI-Hinweis "Entferne erst die andere Person, dann kannst du den Garten löschen". Simplifiziert den Flow; Owner-Transfer ist Post-MVP. Planner erweitert um SQL-Policy auf `garden_members`: `DELETE` von Owner nur wenn `count(members) = 1`.

**Phase-specific security considerations (ASVS categories relevant to Phase 2.5):**

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | yes | Supabase Auth (existing) — keine Änderung |

### Known Threat Patterns for Shared-Garden-Stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| User-A liest User-B's Vereinsregeln (miss-scoped RLS) | Information Disclosure | Member-Check-Policy mit `(select auth.uid())` Wrap; RLS-Test in rls_phase25_shared_garden.sql |
| User fügt sich selbst unberechtigt als Member zu fremdem Garten hinzu | Elevation of Privilege | Direkter INSERT auf `garden_members` ist durch fehlende INSERT-Policy blockiert; einziger Weg ist `consume_invite_code` (SECURITY DEFINER, prüft Code-Validity) |
| Non-Owner erzeugt Invite-Code für fremden Garten | Elevation of Privilege | `create_invite_for_garden` prüft explizit `role = 'owner'` bei v_user im Function-Body |
| Brute-Force gegen 6-stelligen Invite-Code | Spoofing | 31^6 ≈ 887 Mio Möglichkeiten + 24h-Gültigkeit + Ein-Use. Zusätzlich Supabase-Auth-Rate-Limit auf /rpc/* Endpoints (generell). Explizites Rate-Limit für MVP out-of-scope (deferred). |
| Race-Condition: Dritter Member via parallele Consume-Calls | Tampering | BEFORE INSERT Trigger mit `SELECT FOR UPDATE` auf gardens-Row (Pattern 2 Sicherung) |
| SQL-Injection in `consume_invite_code(p_code text)` | Tampering | Postgres Parameter-Binding via PL/pgSQL (nicht string-concat); `upper(p_code)` ist sicher |
| Confused-deputy: SECURITY DEFINER Function mit unsicherem search_path | Elevation of Privilege | Alle SECURITY DEFINER Functions haben `SET search_path = public, pg_temp` (Pattern 3) |
| Old code scheibar nach "Neuen Code erzeugen" noch gültig | Information Disclosure | `create_invite_for_garden` macht `UPDATE … SET consumed_at = now() WHERE … consumed_at IS NULL` VOR dem INSERT des neuen Codes (Pattern 3) |
| NULL-Policy-Window während Migration 003 | Multiple STRIDE | Single-Transaction-Migration (Pattern 1) + Staging-Test vor Production |

**Defense-in-Depth-Spezifika:**

- RLS + SECURITY DEFINER: auch wenn eine Policy fehlschlägt, ist der einzige Mutation-Pfad `garden_members` via `consume_invite_code` der nur authentifizierte User erlaubt (GRANT auf `authenticated` role).
- Client-seitiger Owner-Check in `(app)/settings/garden.tsx` (Button "Person einladen" nur für Owner sichtbar) ist UX, nicht Security — Server-Check ist der Boundary.
- `profiles.display_name` ist user-settable — kein XSS-Filter im DB-Layer, aber React/React Native rendern als Text (kein HTML). Kein Injection-Risiko.
