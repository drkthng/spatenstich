### CR-01: ai_jobs/ai_results member-scope isolation

**Files modified:** `supabase/migrations/20260423000007_ai_jobs_created_by_isolation.sql`
**Commit:** `c02912e`
**Applied fix:** Neue Migration 007 härtet die drei RLS-Policies:
- `ai_jobs_creator_insert` (ersetzt `ai_jobs_member_insert`): WITH CHECK zusätzlich `created_by_user_id = auth.uid()`
- `ai_jobs_creator_read` (ersetzt `ai_jobs_member_read`): USING zusätzlich `created_by_user_id = auth.uid()`
- `ai_results_creator_read` (ersetzt `ai_results_member_read`): USING joint über `ai_jobs.created_by_user_id` via EXISTS (ai_results hat keine eigene Creator-Spalte)

`is_garden_member(garden_id)` bleibt als Defense-in-Depth erhalten. Post-migration-invariant-DO-Block verifiziert, dass alle drei Policies existieren.

**Status:** fixed (DB-RLS-change, requires human `supabase db push --linked`)

### WR-01: `loadMembers` profiles embed fails at runtime

**Files modified:** `supabase/migrations/20260423000008_profiles_member_visible.sql`
**Commit:** `500acfc`
**Applied fix:** Neue SECURITY DEFINER Helper-Function `public.are_co_members(uuid, uuid)` analog zu `is_garden_member` (vermeidet RLS-Rekursion). Zusätzliche SELECT-Only-Policy `profiles_co_member_visible` öffnet profile-Rows für User, die einen gemeinsamen Garten haben. `profiles_own` FOR ALL bleibt unverändert — UPDATE/DELETE bleibt id-basiert.

Trade-off im Migrations-Kommentar dokumentiert: Policy öffnet die ganze Row, nicht nur display_name. Akzeptabel, weil plz/klimazone im geteilten gardens-Row ohnehin sichtbar sind und der Client-Select explizit nur display_name holt.

**Status:** fixed (DB-RLS-change, requires human push)
