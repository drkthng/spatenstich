### Task 02.5-04-03: Root-Layout Bootstrap-useEffect (Commit `60404df`)

**Zweck:** Defense-in-Depth (D-12) — activeGardenId wird serverseitig wiederhergestellt wenn Store leer aber Identity vorhanden.

Modifiziert: `app/app/_layout.tsx`
- Imports: `useAuthStore` + `ensureDefaultGardenForUser`
- `GuardedStack` liest `mode`, `activeGardenId`, `setActiveGarden` aus `useAuthStore`
- `useEffect` mit Dependency-Array `[identity, mode, activeGardenId, setActiveGarden]`:
  - Skip wenn `!identity || mode !== 'account' || activeGardenId`
  - Ruft `ensureDefaultGardenForUser()` (server-idempotent RPC)
  - Pipet Ergebnis durch `setActiveGarden(gardenId)` (cancelled-cleanup verhindert setState post-unmount)
  - RPC-Failure: `__DEV__`-guarded `console.warn`, non-fatal — UI-Screens zeigen per-screen-errors
- Existing Redirect-Logik (`identity === null && !inAuthGroup` + `identity !== null && !inAppGroup`) unverändert

**Covers:** (a) post-deploy signUp ohne migrate-flow, (b) v0 persist-blob rehydrate ohne activeGardenId, (c) pathological missed migration-seed, (d) **post-delete-garden (D-16) re-provisioning** nach User hat seinen einzigen Garten gelöscht → Next-Screen-Mount provisioniert einen neuen default.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] expo-clipboard version mismatch mit Expo SDK 53**
- **Gefunden während:** Task 02.5-04-02 (Clipboard-Import in settings/garden.tsx)
- **Issue:** `pnpm -C app add expo-clipboard` (ohne Version-Pin) installierte `^55.0.13`. SDK-55-Expo-Pakete sind nicht mit SDK 53 (Phase 01 Constraint) kompatibel — Metro-Auflösung hätte gebrochen.
- **Fix:** `pnpm -C app remove expo-clipboard && pnpm -C app add expo-clipboard@~7.0.0` → finale `~7.0.1` (SDK 53 peer).
- **Files modified:** `app/package.json`, `pnpm-lock.yaml`
- **Commit:** `5e1eb30` (mit Task 02)

### Intentional Plan-Deviation

**2. `garden.member.owner_badge` / `member_badge` → `garden.owner_badge` / `member_badge`**
- **Grund:** Plan-Text nutzte `t('garden.member.owner_badge')` und `t('garden.member.member_badge')`, aber Plan 01's de.json hat die Keys auf `garden.owner_badge` / `garden.member_badge` (top-level in garden) gelegt. UI verwendet `t('garden.owner_badge')` + `t('garden.member_badge')` (passt zu Plan 01), nicht die Plan-04-text-Variante.
- **Impact:** Keiner — Keys sind vorhanden und korrekt aufgelöst. Kein Deviation-Report nötig, aber zur Klarheit dokumentiert.

**3. Transfer section_title `garden.transferOwnership.section_title` → `garden.owner_badge`**
- **Grund:** Plan-Text referenziert `t('garden.transferOwnership.section_title') ?? 'Besitzer'`, der Key existiert aber nicht in de.json (Plan 01 hat section-title nicht definiert). UI nutzt `t('garden.owner_badge')` ("Besitzer") als Section-Header — semantisch identisch zum vorgeschlagenen Fallback-String.
- **Impact:** Keiner — User sieht "BESITZER" als Section-Header vor dem Transfer-Button, exakt wie Plan-Text als Fallback vorsah.

Keine Rule-4 Architectural Changes, keine Deferrals an Plan 05.

## Handoff to Phase 3 (Offline & Sync)

**Phase 3 can now assume:**
1. `useAuthStore.getState().activeGardenId` ist nach einem erfolgreichen signIn immer non-null (bootstrap garantiert es, idempotent auch post-delete-garden)
2. UI-Flows für 2-User-Garten sind komplett (Invite, Join, Member-Mgmt, Transfer, Delete) — Phase 3 baut auf dem bestehenden member-check RLS auf
3. LWW-Label-Pattern etabliert in `settings/garden.tsx` — Phase 3 Sync-Status-Indikatoren können dasselbe `formatRelative(iso)` pattern nutzen
4. `garden_id` kommt jetzt per store-pull in `vereinsregelnRepo` + `enqueueAiJob` (Plan 03) — Phase 3's outbox-queue muss dies respektieren (outbox rows referenzieren garden_id, nicht user_id)

- **Garden-Rename-Flow:** Nicht implementiert in Phase 2.5 MVP. `updateGarden` existiert aber wird nur in `migrateLocalToAccount` für plz/klimazone/archetype aufgerufen. UI-Edit für Name kommt ggf. in Phase 5 (Plan-Editor) oder später.

## Known Stubs
