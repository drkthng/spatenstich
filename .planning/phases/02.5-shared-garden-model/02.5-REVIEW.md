## Summary

Die Phase 02.5 liefert das Shared-Garden-Modell (Owner/Member, Einladungs-Codes, D-16 Besitzer-Rechte) über vier Migrationen (003–006), fünf neue/erweiterte Repos, einen neuen `(auth)/join-by-code`-Screen und einen überarbeiteten `settings/garden`-Screen. Tests sind umfangreich: 10 SQL-Dateien für Postgres-Seite (RLS, Member-Limit, Invite-Flow, D-16) und drei Jest-Suites für die Client-Repos.

**Stärken:**
- RLS-Architektur solide: `gardens`, `garden_members`, `invite_codes` haben RLS enabled + KEINE Client-INSERT-Policy → Inserts laufen ausschließlich über SECURITY DEFINER RPCs (`ensure_default_garden_for_user`, `consume_invite_code`, `create_invite_for_garden`). Defense-in-Depth.
- Alle fünf SECURITY DEFINER RPCs haben `SET search_path = public, pg_temp` (bzw. `public, extensions, pg_temp` nach Migration 005). Kein Schema-Hijacking möglich.
- Migration 004 eliminiert die RLS-Rekursion sauber über zwei `STABLE SECURITY DEFINER` Helfer (`is_garden_member`, `is_garden_owner`) mit `REVOKE ALL ... FROM PUBLIC` + `GRANT EXECUTE ... TO authenticated`.
- D-16-Pfade (delete_garden, transfer_ownership) sind atomar und korrekt: atomarer Role-Swap + Member-Count-Guard + Self-Transfer-Guard + Target-Must-Be-Member-Guard. Fehler-Codes (42501/P0003/P0004/P0005) werden konsistent von Postgres → Client-Repo-Typed-Errors → UI i18n gemappt.
- Atomic-Tail-Invariant in `migrateLocalToAccount` korrekt erhalten (Storage.delete strikt nach allen Supabase-Writes).

**Hauptsorgen:**
- **Kritisch:** `ai_jobs`/`ai_results` prüfen jetzt nur `is_garden_member(garden_id)`, nicht mehr `created_by_user_id = auth.uid()`. Ein Member (nicht-Owner) kann AI-Jobs im Namen des anderen Mitglieds einfügen und lesen. Siehe CR-01.
- **Warnungen** betreffen vor allem: Profile-Embed ohne RLS-Abdeckung für `profiles` (WR-01), einen Race in der `activeGardenId`-Invariante bei mode-Wechsel (WR-02), fehlenden `displayName`-Längen-/Sanitation-Guard (WR-03), `invite_codes` wird bei `gardens`-DELETE orphaned im Zeitpunkt des Deletes erst über CASCADE gelöscht — unkritisch, aber potentiell ein Problem beim `transfer_ownership` (WR-05), und toten Code-Zweig in `settings.tsx` (WR-07).

## Critical Issues

### CR-01: `ai_jobs`/`ai_results` member-scope lässt Cross-User-Insert/Read im geteilten Garten zu

**File:** `supabase/migrations/20260423000003_shared_garden.sql:229-236`, `supabase/migrations/20260423000004_fix_rls_recursion.sql:102-112`

**Issue:**
Die Policies `ai_jobs_member_insert`, `ai_jobs_member_read` und `ai_results_member_read` prüfen nur, dass der Aufrufer Member des Gartens ist — aber nicht, dass `created_by_user_id = auth.uid()`. In einem 2-Member-Garten kann User B also:

1. Einen `ai_jobs`-Insert mit `created_by_user_id = <user_a>`, `garden_id = <shared>` absetzen (Policy ist `WITH CHECK (public.is_garden_member(garden_id))` — passt für B, aber `created_by_user_id` ist damit nicht constraint). Das verbrennt User A's AI-Budget (50 Calls/Tag Soft-Limit, 200 Hard — laut CLAUDE.md), weil der Edge-Function-Consumer die Job-Row als "A's" liest.
2. Die `payload` fremder Jobs (z. B. Foto-Analysen von User A mit sensiblen Geo-Daten) lesen. Das ist auch im Shared-Garden ein Problem: Der Edge-Consumer speichert u. U. Foto-URLs + Claude-Responses, die nicht für das andere Mitglied gedacht sind.

Threat-Impact: Budget-DoS + implizite Daten-Exfiltration zwischen den beiden Nutzern (z. B. Dirk vs. Frau — laut Memory `feedback_shared_garden.md` zwei gleichberechtigte User, aber AI-Jobs sind trotzdem individual-scoped).

**Fix:**
`WITH CHECK` um den `created_by_user_id`-Guard erweitern (INSERT) und die SELECT-Policy ebenfalls auf den Ersteller binden — oder explizit dokumentieren, dass Jobs/Results geteilt sein sollen und den Budget-Guard auf Garten-Ebene verschieben.

```sql
-- Migration 007 (new)
DROP POLICY IF EXISTS "ai_jobs_member_insert" ON public.ai_jobs;
CREATE POLICY "ai_jobs_member_insert" ON public.ai_jobs FOR INSERT
  WITH CHECK (
    public.is_garden_member(garden_id)
    AND created_by_user_id = (select auth.uid())
  );

-- Optional (stricter): caller may only read own jobs.
DROP POLICY IF EXISTS "ai_jobs_member_read" ON public.ai_jobs;
CREATE POLICY "ai_jobs_member_read" ON public.ai_jobs FOR SELECT
  USING (
    public.is_garden_member(garden_id)
    AND created_by_user_id = (select auth.uid())
  );

-- Same for ai_results.
```

Alternativ — wenn geteilte Sichtbarkeit beabsichtigt ist — muss das Phasen-Dokument (D-02 / Threat-Model) diese Entscheidung explizit festhalten und ein Test `ai_jobs_created_by_isolation.sql` die aktuellen Policies bestätigen.

### WR-01: `loadMembers` joint `profiles!inner` — RLS auf `profiles` ist id-basiert, nicht member-aware

**File:** `app/src/lib/gardenRepo.ts:112-128`

**Issue:**
`gardenRepo.loadMembers` macht `select('... profile:profiles!inner(display_name)')`. Die RLS-Policy für `profiles` ist aus Migration 002 (siehe `supabase/migrations/20260419000002_profiles.sql` — `using (auth.uid() = id)`) — also reine User-Isolation. Damit sieht User A nur sein EIGENES Profile — NICHT das von User B, obwohl beide Member des gleichen Gartens sind. Das `!inner`-Join filtert B-Rows deshalb im Ergebnis weg (inner join ohne matching profile = Zeile fällt raus). User A bekommt in `members`-Liste dann nur sich selbst statt 2 Mitglieder.

Das erklärt, warum die Tests nur mit Superuser/service_role-Scenarios laufen; die reale Runtime bricht die "2 Member sichtbar"-Semantik.

**Fix:**
Entweder (a) eine `profiles_member_visible`-RLS-Policy ergänzen, die `profiles` an Co-Member im gleichen Garten offenlegt, oder (b) eine SECURITY DEFINER RPC `get_garden_members_with_names(p_garden_id)` einführen, die im Helper-Body auf `profiles` joint (analog `is_garden_member`). Option (b) ist konsistent mit dem "nur-RPC-Schreibpfad"-Muster der Migration.

```sql
-- Option (a) — minimal policy:
CREATE POLICY "profiles_member_visible" ON public.profiles FOR SELECT
  USING (
    id = (select auth.uid())
    OR EXISTS (
      SELECT 1 FROM public.garden_members gm1
      JOIN public.garden_members gm2 ON gm1.garden_id = gm2.garden_id
      WHERE gm1.user_id = (select auth.uid()) AND gm2.user_id = profiles.id
    )
  );
```

### WR-02: Bootstrap-useEffect hat keine Reentrancy-Schutz (D-12) — potentieller Race mit Sign-Out/Sign-In-Wechsel

**File:** `app/app/_layout.tsx:68-83`

**Issue:**
Der `GuardedStack.useEffect` startet `ensureDefaultGardenForUser()` sobald `identity && mode === 'account' && !activeGardenId`. Das `cancelled`-Flag verhindert zwar den `setActiveGarden`-Call nach Unmount, **nicht aber die RPC selbst**. Szenarien, die das RPC doppelt feuern lassen:

1. User macht Logout → `clearAuth()` (`activeGardenId: null`, `mode: null`) → `identity` flippt auf `null` → effect re-runs mit `identity !== null` → nichts passiert. Login danach → `activeGardenId` bleibt `null` → RPC feuert. OK.
2. **Aber:** Wenn User A während Phase-2.5 `deleteGarden()` im Screen aufruft, setzt der Screen `setActiveGarden(null)` und navigiert nach `/(auth)`. Der Guard redirectet aber mit `identity !== null && !inAppGroup`  wieder zurück zu `/(app)`. In der Zwischenzeit feuert das Effect das RPC erneut und erzeugt "Mein Garten" direkt nach dem Delete. Das ist per D-16 so gewollt, aber die Sequenz in `settings/garden.tsx:202-218` navigiert nach `/(auth)` — das ist inkorrekt (siehe INFO IN-01).
3. Beim simultanen Tab-Wechsel (Web) oder Hot-Reload kann das Effect mehrfach laufen, bevor die erste `setActiveGarden`-Zustandsänderung durch Zustand propagiert — der Server ist idempotent, aber man triggert unnötige RPC-Calls.

**Fix:**
Eine `useRef<boolean>(false)`-In-Flight-Flag einführen, die das gleichzeitige Feuern blockiert:

```tsx
const inFlight = React.useRef(false);
React.useEffect(() => {
  if (!identity || mode !== 'account' || activeGardenId || inFlight.current) return;
  inFlight.current = true;
  let cancelled = false;
  (async () => {
    try {
      const gardenId = await ensureDefaultGardenForUser();
      if (!cancelled) setActiveGarden(gardenId);
    } catch (e) {
      if (__DEV__) console.warn('ensureDefaultGardenForUser failed', e);
    } finally {
      inFlight.current = false;
    }
  })();
  return () => { cancelled = true; };
}, [identity, mode, activeGardenId, setActiveGarden]);
```

### WR-03: `display_name` wird ohne Längen-/Sanitation-Limit geschrieben — DoS / UI-Bruch möglich

**File:** `app/src/lib/migrateLocalToAccount.ts:99-106`, `supabase/migrations/20260423000003_shared_garden.sql:100` (ADD COLUMN text)

**Issue:**
Das `display_name`-Feld ist Postgres `text` ohne CHECK-Constraint. `migrateLocalToAccount` leitet den Wert aus `email.split('@')[0]` ab (bis zu 254 Zeichen möglich nach RFC 5321 local-part). `settings/garden.tsx:283` rendert `{m.displayName ?? '?'}` direkt — RN Text wrapped lang, aber die Owner-Badge-Row wird bei 254-Zeichen-Namen unbedienbar. Auch: UTF-8-Zeichen wie Zero-Width-Joiner oder RTL-Overrides würden die RN-Layout brechen.

Kein Security-Bypass (RN escapt XSS automatisch — siehe INFO IN-02), aber:
- Budget-/Quota-Impact gering (Text-Feld im DB billig)
- UI-Bruch realistisch bei Power-User-Manipulation des `profiles.display_name` per Direct-Upsert (saveProfile akzeptiert beliebige Strings in `displayName`)

**Fix:**
- DB-Constraint: `ALTER TABLE public.profiles ADD CONSTRAINT display_name_len CHECK (char_length(display_name) BETWEEN 1 AND 40);`
- Client-Side Limit + Trim in `profileRepo.saveProfile` (max 40 chars, `trim()`, unicode NFC-normalize).
- Für Migration: `split_part(u.email, '@', 1)` bereits in Migration 003 Zeile 103 — dort einen `left(split_part(...), 40)` verwenden.

### WR-04: `consume_invite_code` wirft `P0002` — aber `no_data_found`-Semantik lässt sich mit echtem NO_DATA_FOUND vermischen

**File:** `supabase/migrations/20260423000003_shared_garden.sql:344`, `app/src/lib/inviteCodeRepo.ts:42-45`

**Issue:**
SQLSTATE `P0002` entspricht dem PL/pgSQL-built-in `no_data_found` (wird automatisch von `SELECT INTO STRICT` ausgelöst, wenn das Statement keine Zeile liefert). Wenn ein zukünftiger Trigger auf `invite_codes` oder `garden_members` ein internes `SELECT INTO STRICT` ohne Match wirft, bekommt der Client ebenfalls `P0002` und mappt fälschlicherweise auf `"Code ungültig oder abgelaufen"`. Das ist ein latenter Bug, der sich erst bei zukünftigen Trigger-Erweiterungen manifestiert.

**Fix:**
Einen eigenen Custom-SQLSTATE wählen (Postgres empfiehlt für User-Defined den `P`-Block, aber nicht kollidierend mit Built-Ins). Z. B. `P9002` (kein Built-In). Identische Behandlung nötig für `garden_has_members` (`P0003` — `no_data_found` ist `P0002`, `P0003` ist `too_many_rows`).

Alternativ: Bei `no_data_found`-Catching im Test stattdessen auf `sqlerrm LIKE '%invite_invalid_or_expired%'` prüfen, wie es `delete_garden_forbidden_if_members.sql:59` bereits korrekt macht. Der `invite_code.sql:43` Test fängt `no_data_found` was zufällig funktioniert, aber das ist fragile coupling.

```sql
-- Custom SQLSTATE (recommended ranges: P9xxx or use ERRCODE_RAISE_EXCEPTION 'P0001' + custom text):
RAISE EXCEPTION 'invite_invalid_or_expired' USING ERRCODE = 'P9001';
RAISE EXCEPTION 'garden_has_members' USING ERRCODE = 'P9003';
RAISE EXCEPTION 'cannot_transfer_to_self' USING ERRCODE = 'P9004';
RAISE EXCEPTION 'target_not_member' USING ERRCODE = 'P9005';
```

Clients müssen parallel nachziehen.

### WR-05: `transfer_ownership` aktualisiert `gardens.created_by_user_id` — das ist semantisch falsch (Audit-Verlust)

**File:** `supabase/migrations/20260423000003_shared_garden.sql:500-503`

**Issue:**
Der `transfer_ownership`-Body führt einen UPDATE auf `gardens.created_by_user_id = p_to_user_id` aus. `created_by_user_id` ist per Namenskonvention und D-14 audit-invariant — "wer hat die Row erstellt". Der neue Owner hat aber den Garten NICHT erstellt, er hat ihn ÜBERNOMMEN. Das verfälscht Auditing + bricht die semantische Contract mit anderen Tabellen (`vereinsregeln.created_by_user_id`, `ai_jobs.created_by_user_id`).

Die korrekte Information (wer ist aktueller Owner) ist bereits in `garden_members.role` abgebildet — kein Redundanz-Update nötig. Der `updated_by_user_id = v_user` ist ok (wer hat die Ownership-Änderung ausgelöst).

**Fix:**
`gardens.created_by_user_id` beim Transfer NICHT aktualisieren. Nur `updated_by_user_id` stempeln:

```sql
UPDATE public.gardens
SET updated_by_user_id = v_user
WHERE id = p_garden_id;
```

Wenn zusätzlich eine "aktueller Owner"-Spalte an `gardens` gewollt ist, eine separate `current_owner_user_id`-Column einführen, nicht `created_by` hijacken.

### WR-06: `settings/garden.tsx` `handleCreateInvite` klassifiziert `23514` als "garden ist voll" — aber `23514` kann auch vom 2-Member-Trigger kommen

**File:** `app/app/(app)/settings/garden.tsx:128-131`

**Issue:**
`create_invite_for_garden` selbst kann nie `23514` werfen (es führt nur INSERT auf `invite_codes`, und dort gibt's keinen CHECK mit `23514`). Das Code-Mapping ist tot. Gleichzeitig fehlt das Mapping für `P0001` (code_generation_failed nach 10 Collision-Retries — siehe `shared_garden.sql:310`). Das ist zwar astronomisch unwahrscheinlich, aber würde als `error_generic` kommen, was ok ist.

Kritischer: Die "bereits 2 Mitglieder"-Bedingung sollte **vor** dem Create-Call schon im UI abgefangen werden — der Screen macht das bereits korrekt via `members.length < 2`-Gating (Zeile 329), aber falls gleichzeitig eine Race stattfindet (zweiter Member tritt bei, während Owner den Invite-Knopf drückt), sollte der RPC selbst einen dedizierten Fehler werfen.

**Fix:**
- Entfernen des `23514`-Mappings in `handleCreateInvite` (toter Code, bewusster Fehlermapping-Fallback).
- Im RPC ein `garden_already_full` Check ergänzen:

```sql
-- In create_invite_for_garden, vor dem UPDATE der alten Codes:
SELECT count(*) INTO v_member_count FROM public.garden_members WHERE garden_id = p_garden_id;
IF v_member_count >= 2 THEN
  RAISE EXCEPTION 'garden_already_full' USING ERRCODE = 'P9006';
END IF;
```

Alternativ: Das `23514`-Mapping beibehalten und durch ein echtes CHECK auf `invite_codes` unterlegen (z. B. "nur ausstellbar wenn Garden < 2 Mitglieder"). Aktuell ist das Mapping reine Cargo-Cult.

### WR-07: `settings.tsx` `handleMigrate` — toter Branch-Zweig und generische Fehlermeldung maskiert RPC-Fehler

**File:** `app/app/(app)/settings.tsx:81-87`

**Issue:**
```tsx
if (msg.includes('already in account mode')) {
  setMigError(t('auth.register.error_generic'));
} else {
  setMigError(t('auth.register.error_generic'));
}
```

Beide Branches setzen die identische Meldung — die Verzweigung ist tot. Zusätzlich maskiert der `error_generic`-String Fehler wie `migration_partial_garden_seed`, `migration_partial_profile`, `migration_partial_garden_metadata`, `migration_partial_vereinsregeln` — der User sieht nur "Registrierung fehlgeschlagen. Bitte prüfen ob diese E-Mail bereits verwendet wird.", obwohl die Registrierung ERFOLGREICH war und nur ein Folge-Schritt fehlschlug. Das ist user-feindlich (User denkt, er muss andere Email nehmen, obwohl Retry genügt) und diagnostisch wertlos.

**Fix:**
Spezifische Error-Map mit Retry-Hinweisen:

```tsx
} catch (e) {
  const msg = e instanceof Error ? e.message : String(e);
  if (msg.startsWith('migration_partial_')) {
    // Signup war erfolgreich — User zu login/retry leiten.
    setMigError('Account angelegt, aber Datenübertragung fehlgeschlagen. Bitte erneut versuchen.');
  } else if (msg.includes('already in account mode')) {
    setMigError('Du bist bereits angemeldet.');
  } else {
    setMigError(t('auth.register.error_generic'));
  }
}
```

Zusätzlich: bei `migration_partial_*`-Fehlern ist der User bereits bei Supabase registriert, aber `authStore` ist noch im `local`-Mode. Nächster Retry wird dann am `already in account mode`-Guard scheitern (wenn signUp duplicate-email erkennt) — die Rollback-Logik ist unvollständig. Empfohlen: Bei `migration_partial_*` den authStore-Mode trotzdem auf `account` flippen und ab da Retry über einen separaten "Daten nachladen"-Button anbieten.

### IN-03: `enqueueAiJob` setzt `updated_by_user_id` nicht — Trigger fängt das auf, aber inkonsistent mit Pattern 6

### IN-04: `join-by-code` Crockford-Filter entfernt `0` aber nicht `O/I/L/U`

**File:** `app/app/(auth)/join-by-code.tsx:32-39`, `app/src/lib/inviteCodeRepo.ts:31-33`

**Issue:**
Der Client-Side-Filter `/[^A-Z1-9]/g` entfernt `0` (nicht in `1-9`), lässt aber `O`, `I`, `L`, `U` stehen (Teil von `A-Z`). Der Server generiert die Codes aus `123456789ABCDEFGHJKMNPQRSTVWXYZ` (ohne 0/O/I/L/U), also wird ein valider generierter Code niemals diese Zeichen enthalten. Aber wenn ein User einen Code tippt und aus Versehen "O" statt "0" oder "I" statt "1" eingibt, wird er nicht korrigiert — der RPC-Lookup schlägt fehl und der User bekommt `"Code ungültig oder abgelaufen"`. Das ist suboptimales UX: Crockford's Idee ist genau, dass diese Zeichen ineinander konvertiert werden.

Der `inviteCodeRepo.test.ts:79-94` dokumentiert das bewusst — der Server ist dafür zuständig. Aber der Server macht in `consume_invite_code.sql:338` nur `upper(code)` — auch keine Crockford-Normalisierung.

**Fix:**
Entweder Client oder Server soll Crockford-Aliases normalisieren:

```ts
// normalizeCode in inviteCodeRepo.ts:
function normalizeCode(raw: string): string {
  return raw
    .trim()
    .toUpperCase()
    .replace(/O/g, '0')  // Wait — 0 is NOT in the alphabet. O should map to nothing.
    // Crockford canonical: O/I/L/U aren't in the alphabet. Best option: strip them + 0.
    .replace(/[0OILU]/g, '')
    .replace(/[^A-Z1-9]/g, '')
    .slice(0, 6);
}
```

Oder die Alphabet-Wahl überdenken (wenn User-Input-Resilienz wichtiger ist, Crockford definiert `O→0, I→1, L→1, U→V`-Mapping; dann muss aber das Server-Alphabet 0/1 akzeptieren).

### IN-05: `migrateLocalToAccount` Sentry-User-Set fehlt — Phase-2-Sentry-Gating hier nicht repliziert

### IN-06: Test-SQL-Files nutzen `as any`-artige `reset role; set local role authenticated;` ohne Kommentar, dass das in Production-DB nicht passieren darf
