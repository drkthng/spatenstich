## RLS Policy Rotation

Migration 003 dropped die alten `*_own`-Policies (USING `auth.uid() = user_id`) und installierte Member-Check-Policies. Migration 004 re-rotierte dann alle Policies, die Subquery-Joins auf `garden_members` enthielten (wegen Infinite-Recursion 42P17), auf Helper-Aufrufe:

| Table            | Policy                                     | Final USING-Expression (after Migration 004) |
|------------------|--------------------------------------------|-----------------------------------------------|
| `gardens`        | `gardens_member_select` (SELECT)           | `public.is_garden_member(id)`                |
| `gardens`        | `gardens_member_update` (UPDATE)           | `public.is_garden_member(id)` + WITH CHECK same |
| `gardens`        | `gardens_owner_delete` (DELETE)            | `public.is_garden_owner(id)` (Defense-in-Depth â€” primary path ist delete_garden RPC) |
| `garden_members` | `garden_members_member_select` (SELECT)    | `public.is_garden_member(garden_id)`         |
| `garden_members` | `garden_members_owner_update` (UPDATE)     | `public.is_garden_owner(garden_id)`          |
| `garden_members` | `garden_members_self_or_owner_delete` (DELETE) | `user_id = (select auth.uid()) OR public.is_garden_owner(garden_id)` |
| `garden_members` | (no INSERT policy)                         | Deny-by-default â†’ nur consume_invite_code RPC (SECURITY DEFINER) kann INSERT |
| `invite_codes`   | `invite_codes_member_select` (SELECT)      | `public.is_garden_member(garden_id)`         |
