## Garten-Modell & Profil-Scope

### Q1: PLZ + Klimazone + Archetyp — was bleibt per-User, was wandert auf den Garten?

| Option | Description | Selected |
|--------|-------------|----------|
| Alles auf Garten (PLZ/Klimazone/Archetyp) | Recommended. Ein Kleingarten = eine PLZ/Klimazone. Archetyp = Garten-Charakter. profiles behält nur id + display_name. | ✓ |
| Hybrid: PLZ/Klimazone auf Garten, Archetyp per User | Archetyp als persönliche Sicht. Mehr Kombinations-Komplexität downstream. | |
| Alles bleibt per-User | gardens hat nur id+name+owner. Konfliktpotenzial bei abweichender Klimazone. | |

**User's choice:** Alles auf Garten (PLZ/Klimazone/Archetyp)

### Q2: Welche Phase-2-Tabellen bekommen `garden_id`-FK?

| Option | Description | Selected |
|--------|-------------|----------|
| vereinsregeln + ai_jobs + ai_results | Recommended. Konsistent mit downstream. Vision-Calls für beide Member sichtbar. | ✓ |
| Nur vereinsregeln | ai_jobs/results bleiben per-user — Inkonsistenz. | |
| Keine | Phase-2-Tabellen bleiben per-User — widerspricht Shared-Garden-Prinzip. | |

**User's choice:** vereinsregeln + ai_jobs + ai_results

### Q3: Felder der neuen `gardens`-Tabelle?

| Option | Description | Selected |
|--------|-------------|----------|
| Minimal: id, name, created_at, updated_at, created_by_user_id | Recommended. Inhaltliche Felder aus profiles. Owner via created_by. | ✓ |
| Rich: zusätzlich masse (L×B), form, hausnummer, beschreibung | Parzellen-Metadaten — Phase 4 kommt eh damit. | |
| Maximal-minimal (nur id + created_at) | Keine Identifikation, verwirrt. | |

**User's choice:** Minimal: id, name, created_at, updated_at, created_by_user_id

### Q4: "Zuletzt bearbeitet von <Name>" — woher kommt der Anzeige-Name?

| Option | Description | Selected |
|--------|-------------|----------|
| Neue Spalte `display_name` in `profiles` | Recommended. Einmal gesetzt beim Onboarding, global nutzbar. | ✓ |
| Auf `garden_members` als `display_name_in_garden` | Pro Garten anderer Name — Flexibilität aktuell unnötig. | |
| Kein display_name — nur E-Mail / User-ID | Hässlich, im Lokal-Modus kein E-Mail. | |

**User's choice:** Neue Spalte display_name in profiles

### Q5 (Follow-up): FK-Struktur — user_id ersetzen oder beides behalten?

| Option | Description | Selected |
|--------|-------------|----------|
| `user_id` ersetzen durch `garden_id` | Schlank, aber Audit-Info verloren. | |

**User's choice:** Other → "du entscheidest"
**Claude's Discretion decision:** Beides behalten (`garden_id` + `updated_by_user_id`), da SC4 sonst nicht erfüllbar.

## Invite-Code-Flow

### Q1: Wer darf einen Invite-Code erzeugen?

| Option | Description | Selected |
|--------|-------------|----------|
| Nur Owner (created_by_user_id) | Recommended. Klare Rechtefrage. Für 2-User-MVP ausreichend. | ✓ |
| Jeder Member | Im MVP sind beide Plätze belegt — pointless. | |
| Owner + explizit delegiert | Over-engineered bei harter 2-Member-Grenze. | |

**User's choice:** Nur Owner (created_by_user_id)

### Q2: Codeform + Delivery?

| Option | Description | Selected |
|--------|-------------|----------|
| Klar-lesbar (z.B. 'HELLO7') + Copy + Share | Recommended. Am Telefon diktierbar. Kein 0/O/1/I/l. | ✓ |
| Numerisch 6-stellig ('483201') + Copy + Share | SMS-TAN-Style, weniger Entropie (10^6). | |
| UUID-Link (deep-link) | Braucht Universal-Links-Setup — andere Phase. | |

**User's choice:** Klar-lesbar (kein 0/O/1/I/l), 6 Zeichen, Copy + Share-Sheet

### Q3: Gültigkeit + Use-Count?

| Option | Description | Selected |
|--------|-------------|----------|
| 24h gültig, Ein-Use | Recommended. Sicher, einfach. | ✓ |
| 7 Tage gültig, Ein-Use | Entspannter, etwas größeres Missbrauchsfenster. | |
| Bis revoked, Mehrfach-Use | Mehrfach-Use pointless bei 2-Member-Cap. | |

**User's choice:** 24h gültig, Ein-Use (consumed bei Beitritt)

### Q4: Wo im UI?

| Option | Description | Selected |
|--------|-------------|----------|
| Settings → 'Mein Garten' → 'Person einladen'; Beitritt via 'Code eingeben' in Onboarding/Settings | Recommended. Klare Trennung Owner/Joiner. | ✓ |
| Ein Screen mit Tabs 'Einladen' / 'Beitreten' | Dead-Space pro Rolle. | |
| Onboarding-Verzweigung 'Neuen Garten' vs 'Beitreten' | Scope-Creep — Onboarding ist schon released. | |

**User's choice:** Settings → 'Mein Garten' → 'Person einladen'; Beitritt via 'Code eingeben' in Onboarding ODER Settings

### Q5 (Follow-up): Revokation?

| Option | Description | Selected |
|--------|-------------|----------|
| Auto: 'Neuen Code erzeugen' invalidiert alten | Recommended. Ein aktiver Code pro Garten. | ✓ |
| Expliziter 'Widerrufen'-Button | Mehr Kontrolle, eigene UI. | |
| Nur Ablauf (24h), kein Widerruf | Minimal — 24h-Fenster manchmal zu lang. | |

**User's choice:** Auto — 'Neuen Code erzeugen' invalidiert den alten
