# Phase 2.5 Plan 01: Shared Garden Foundation (Types + i18n + Test-Stubs) Summary

**Fundament für Phase 2.5 gelegt — Domain-Typen, i18n-Strings, Requirements und Test-Verification-Targets stehen; Plan 02 (Migration 003) und Plan 03 (Repos) haben jetzt konkrete Contract-Ziele.**

## Overview

Plan 01 hat den zirkulären Abhängigkeits-Knoten zwischen Plan 02 (DB-Migration), Plan 03 (Repos) und Plan 04 (UI) aufgelöst: alle nachfolgenden Pläne können nun gegen feste Typen (`Garden`, `GardenMember`, `InviteCode`, `GardenRole`, `LocalProfile`), feste i18n-Strings (inkl. der 14 D-16 Owner-Rights-Keys für `garden.transferOwnership.*` + `garden.delete.*`) und feste Test-Contracts (4 SQL-Stubs + 3 Jest-Stub-Suites) bauen. Kein Runtime-Code, keine DB-Änderungen — reiner Fundament-Plan, wie im Plan-Design vorgesehen.

Die Clean-Cut-Variante aus RESEARCH §9 / PATTERNS §12 wurde umgesetzt: `UserProfile` ist auf Account-Scope geschrumpft (`userId`, `mode`, `displayName`, `createdAt`, `updatedAt`), `LocalProfile extends UserProfile` preserviert PLZ/Klimazone/Archetyp für den Lokal-Modus (D-13), und `Garden` trägt diese Felder nun am Garten-Level (D-01).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Test-Framework-Import: vitest → jest**

- **Gefunden während:** Task 02.5-01-05
- **Issue:** Plan-Text `<action>`-Block enthielt `import { describe, it } from 'vitest';`. Das Projekt verwendet aber Jest (`jest-expo` + `ts-jest`, konfiguriert via `app/jest.config.ts`). Vitest ist nicht installiert, der Import hätte zu `Cannot find module 'vitest'`-Fehlern beim Jest-Run geführt.
- **Fix:** Import-Zeile in `gardenRepo.test.ts` + `inviteCodeRepo.test.ts` entfernt. Jest stellt `describe`/`it`/`it.todo` als Globals über `jest-expo` bereit (Pattern identisch zu existierender `vereinsregelnRepo.test.ts`). Kommentar ergänzt, der das Framework dokumentiert.
- **Files modified:** `app/src/lib/__tests__/gardenRepo.test.ts`, `app/src/lib/__tests__/inviteCodeRepo.test.ts`
- **Commit:** `aaa646a`

**2. [Rule 3 - Blocking Issue] `migrateLocalToAccount.ts` Typ-Signatur entblockt Jest-Run**

- **Gefunden während:** Task 02.5-01-05 Verify-Step (Jest-Run auf 3 Ziel-Files)
- **Issue:** Nach der Plan-2-Änderung von `UserProfile` (Entfernung von `plz`/`klimazone`/`archetype` per D-01) schlug `ts-jest` beim Kompilieren der Test-Suite mit `TS2339: Property 'plz' does not exist on type 'Partial<UserProfile>'` fehl. Das blockierte den gesamten Jest-Run und damit Acceptance-Criterium 5.
- **Fix:** Minimaler Typ-Fix — `import type { LocalProfile, VereinsRegel } from '@spatenstich/shared'` statt `UserProfile`, und `Partial<UserProfile>` → `Partial<LocalProfile>` im lokalen Profile-Blob-Cast. Der Funktionskörper bleibt behavioral identisch — der Lokal-Storage-Blob hatte schon immer LocalProfile-Semantik (PLZ/Klimazone/Archetyp), der vorherige Typ-Cast war nur möglich, weil UserProfile diese Felder hatte. Keine Änderung an Logic, Signatur, oder API. Plan 03 wird die Funktion eigenständig um `ensure_default_garden_for_user()`-RPC + gardenId-Stempel erweitern.
- **Files modified:** `app/src/lib/migrateLocalToAccount.ts` (2 Zeilen: Import + Cast-Typ), `app/src/lib/__tests__/migrateLocalToAccount.test.ts` (2 Zeilen: Import + `LOCAL_PROFILE`-Typ-Annotation)
- **Commit:** `aaa646a`
