### 1. `supabase/migrations/20260423000003_shared_garden.sql` (migration, DDL+seed+RLS-refactor)

**Analog 1:** `supabase/migrations/20260419000002_profiles.sql` — Struktur `CREATE TABLE → ALTER TABLE ENABLE RLS → CREATE POLICY → CREATE TRIGGER`.
**Analog 2:** `supabase/migrations/20260416000001_foundation.sql` — Trigger-Function `tg_set_updated_at()` + `CREATE TRIGGER` pro Tabelle.

**Header-Kommentar-Pattern** (20260419000002_profiles.sql:1-4):
```sql
-- Phase 2 / Migration 002 — Profiles + Vereinsregeln + Storage Bucket
-- D-11: profiles table is Account-only; lokal-mode users persist via StorageAdapter (kein DB-Eintrag).
-- D-02: RLS auf allen Tabellen aktiv; auth.uid()-Policy mit WITH CHECK.
-- Referenz: public.tg_set_updated_at() wurde in Migration 001 (20260416000001_foundation.sql) definiert — nicht neu anlegen.
```
Für Migration 003 analog: D-Nummer-Kommentare + explizite Erwähnung der Vorgänger-Migrations und der `tg_set_updated_at`-Wiederverwendung.

**CREATE TABLE + RLS + Policy Block-Pattern** (20260419000002_profiles.sql:6-29):
```sql
-- ────────────────────────────────────────────────────────────────
-- 1. profiles (Account-only: id = auth.users.id)
-- ────────────────────────────────────────────────────────────────
create table public.profiles (
  id          uuid primary key references auth.users(id) on delete cascade,
  plz         text,
  klimazone   smallint check (klimazone between 1 and 7),
  archetype   text check (archetype in (
    'selbstversorger','familien_naschgarten','mix_ausgewogen',
    'zier_erholung','biodiversitaet','kraeuter_apotheker'
  )),
  created_at  timestamptz not null default now(),
  updated_at  timestamptz not null default now()
);

alter table public.profiles enable row level security;

create policy "profiles_own" on public.profiles
  for all
  using (auth.uid() = id)
  with check (auth.uid() = id);

create trigger profiles_updated_at before update on public.profiles
  for each row execute function public.tg_set_updated_at();
```
Diesen kompakten Block 1:1 für `gardens`, `garden_members`, `invite_codes` spiegeln. **Aber:** Policy-Form ist anders — Phase 2.5 nutzt Member-Check statt `auth.uid() = id` und **muss** den `(select auth.uid())`-Wrap verwenden (RESEARCH §Pattern 5 / Supabase Advisor 0003). Beispiel aus RESEARCH §Pattern 5 ist in 02.5-RESEARCH.md:463-473 nachlesbar.

**Foundation-Migration-Trigger-Wiederverwendung** (20260416000001_foundation.sql:91-99):
```sql
-- updated_at-Trigger (DRY)
create or replace function public.tg_set_updated_at()
returns trigger language plpgsql as $$
begin new.updated_at = now(); return new; end $$;

create trigger feature_flags_updated_at before update on public.feature_flags
  for each row execute function public.tg_set_updated_at();
create trigger ai_jobs_updated_at before update on public.ai_jobs
  for each row execute function public.tg_set_updated_at();
```
Migration 003 **ruft `public.tg_set_updated_at()` nur als `CREATE TRIGGER … EXECUTE FUNCTION …` auf** — die Function selbst bleibt aus Migration 001. Zusätzlich neu: `tg_set_updated_by_user_id()` analog dazu (RESEARCH §Pattern 6).

**DO-Block-idempotent-Pattern für pgmq-Queues** (20260416000001_foundation.sql:82-89) — zeigt wie man idempotente Seeds schreibt; Planner kann dasselbe `do $$ begin if not exists (…) then … end if; end $$;`-Muster anwenden, falls irgendein Seed-Step idempotent gehalten werden muss (z.B. wenn Migration auf bereits-migrated Datenbank zweimal läuft).

**Volle Migration-Reihenfolge (10 Schritte):** in 02.5-RESEARCH.md §"Full Migration Template" (Zeilen 702–1098) steht die komplette SQL — der Planner kopiert sie in den Migration-Task als Blueprint. Reihenfolge zwingend:
1. CREATE neue Tabellen (gardens, garden_members, invite_codes)
2. Member-Limit-Trigger
3. profiles.display_name ADD COLUMN + backfill via `split_part(email, '@', 1)`
4. Default-Gärten seed (ein Garten pro bestehender profile)
5. garden_id + updated_by_user_id ADD COLUMN auf vereinsregeln/ai_jobs/ai_results (nullable)
6. Backfill garden_id aus user_id-Lookup → SET NOT NULL
7. RENAME user_id → created_by_user_id (D-06)
8. DROP old `*_own` policies + CREATE new `*_member` policies (atomic)
9. RPCs: `gen_invite_code`, `create_invite_for_garden`, `consume_invite_code`, `ensure_default_garden_for_user`
10. Trigger `tg_set_updated_by_user_id()` + Trigger-Attachments

**Pitfall-Schutz:**
- `profiles.plz/klimazone/archetype` **nicht droppen** in Migration 003 (Pitfall 4 — zwei-Phase-Refactor).
- Policies in **einem** File rotieren (DROP + CREATE unmittelbar hintereinander).
- Alle Policies mit `(select auth.uid())`-Wrap.

```sql
BEGIN;
  -- ── Als User A: JWT-Claims setzen + Rolle wechseln, dann INSERTs ──
  select set_config('request.jwt.claim.sub', 'b9667f2a-7e86-497b-9cff-3ae320c7ff2c', true);
  select set_config('request.jwt.claim.role', 'authenticated', true);
  set local role authenticated;

  -- Profile für User A anlegen
  insert into public.profiles(id, plz, klimazone, archetype)
    values ('b9667f2a-7e86-497b-9cff-3ae320c7ff2c'::uuid, '12043', 4, 'selbstversorger');

  -- Vereinsregel für User A anlegen
  insert into public.vereinsregeln(user_id, source, titel, wert, einheit, ist_bkleingg)
    values ('b9667f2a-7e86-497b-9cff-3ae320c7ff2c'::uuid, 'checklist', 'Heckenhöhe max', 120, 'cm', false);

  -- ── Als User B: JWT-Claim wechseln, SELECTs müssen 0 Zeilen ergeben ──
  select set_config('request.jwt.claim.sub', 'd5fc0937-5e49-4fb5-9740-9940f075fab2', true);

  do $$
  declare cnt int;
  begin
    -- profiles isolation
    select count(*) into cnt
      from public.profiles
     where id = 'b9667f2a-7e86-497b-9cff-3ae320c7ff2c'::uuid;
    if cnt <> 0 then
      raise exception 'RLS breach (profiles): user B sees % rows of user A', cnt;
    end if;
    raise notice 'RLS ok (profiles): user B sieht 0 Zeilen von user A';
    -- ...
  end $$;

ROLLBACK;
```

**Erweiterungen für Phase 2.5:**
- Nach User-A-Inserts: `INSERT INTO gardens (…) RETURNING id INTO ga_id` + `INSERT INTO garden_members (ga_id, user_a, 'owner')` + vereinsregeln-Insert mit `garden_id = ga_id`.
- Zusätzliche DO-Blöcke:
  - `rls ok: B sieht 0 vereinsregeln von Garten A`
  - `rls ok: B sieht 0 ai_jobs von Garten A`
  - `garden_members_limit_2 fires bei 3. Insert` (INSERT 3. member → erwarten `raise exception 'garden_member_limit_exceeded'`)
  - `consume_invite_code by user_B → ga_id zurück, B ist jetzt member` — optional in separatem File `invite_code.sql`.
- `ROLLBACK` am Ende (nicht COMMIT) — identisch zu Phase 2.

**Aufteilungs-Empfehlung (Orchestrator gab 3 Files vor):**
- `rls_member_check.sql` — Isolation pro Tabelle (wie rls_phase2.sql)
- `member_limit.sql` — 2-Member-Trigger

Alle 3 erben dasselbe BEGIN/ROLLBACK-Gerüst aus rls_phase2.sql.

### 4. `app/src/lib/inviteCodeRepo.ts` (RPC-wrapper, request-response)

**Analog:** `app/src/lib/vereinsregelnRepo.ts` (Struktur) + RESEARCH.md §Pattern 3+4 (RPC-Signatur) + bestehender `supabase.auth.signUp`-Style aus `migrateLocalToAccount.ts:54-58` (async/await + error-Unwrap).

**Kein toRow/fromRow — pures RPC-Wrapping.** Datei-Shape-Blueprint:
```typescript
// Invite-Code RPC wrapper — Plan 02.5-XX.
// Account-mode only; lokal-mode wirft.
// Pattern: vereinsregelnRepo.ts (mode-check + supabase-error-unwrap) +
// RESEARCH §Pattern 3 (SECURITY DEFINER RPCs via supabase.rpc).
import { supabase } from './supabase';
import type { AuthMode } from '../stores/authStore';

export async function createInviteForGarden(
  mode: AuthMode,
  gardenId: string,
): Promise<string> {
  if (mode !== 'account') throw new Error('gardens are account-only');
  const { data, error } = await supabase.rpc('create_invite_for_garden', {
    p_garden_id: gardenId,
  });
  if (error) throw error;  // '42501' = not_owner / not_authenticated
  return data as string;   // 6-char code
}

export async function consumeInviteCode(
  mode: AuthMode,
  code: string,
): Promise<string> {
  if (mode !== 'account') throw new Error('gardens are account-only');
  const { data, error } = await supabase.rpc('consume_invite_code', {
    p_code: code.trim().toUpperCase(),
  });
  if (error) throw error;  // 'P0002' = invalid_or_expired
  return data as string;   // garden_id uuid
}

export async function ensureDefaultGardenForUser(): Promise<string> {
  const { data, error } = await supabase.rpc('ensure_default_garden_for_user');
  if (error) throw error;
  return data as string;
}
```
**Error-Mapping** (für UI-Layer): `error.code === '42501'` → "Nicht berechtigt"; `error.code === 'P0002'` → "Code ungültig oder abgelaufen"; `error.code === '23514'` → "Garten ist voll" (Member-Limit). Pattern analog zu vereinsregelnRepo.ts:109 (`if (error) throw error`), nur mit zusätzlicher Code-Klassifikation im UI (join-by-code.tsx).

### RLS Member-Check (alle Policies auf Garten-scoped Tabellen)

### Migration — Atomic-Tail-Pattern

**Source:** `app/src/lib/migrateLocalToAccount.ts:4-14 + 114-117`
**Apply to:** `migrateLocalToAccount.ts` (erweitert)

`storage.delete(*)` läuft **strictly nach** allen Supabase-Side-Effects. Für erweiterten Flow:
1. signUp → throw-on-error
2. rpc('ensure_default_garden_for_user') → throw-on-error
3. read local (read-only)
4. upsert profiles → throw-on-error
5. upsert gardens (PLZ/Klima/Archetyp) → throw-on-error
6. upsert vereinsregeln (mit gardenId) → throw-on-error
7. setAccountMode + setActiveGarden
8. storage.delete('profile') + storage.delete('vereinsregeln')
