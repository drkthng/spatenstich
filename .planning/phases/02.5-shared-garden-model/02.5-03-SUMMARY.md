# Phase 2.5 Plan 03: Data-Layer (Repos + Store + Migration Extension) Summary

**Daten-Schicht für Shared Garden live: 2 neue Repos (gardenRepo + inviteCodeRepo mit D-16 Owner-Rights), 4 erweiterte Bestands-Dateien (profileRepo, vereinsregelnRepo, migrateLocalToAccount, enqueueAiJob), authStore um activeGardenId extended, 22 + 13 + 6 + 7 neue Jest-Tests grün, 109/109 Tests grün, typecheck clean — Plan 04 kann UI gegen echte Repos bauen.**

## Overview

Plan 03 ist die Daten-Schicht-Brücke zwischen der Schema-Migration in Plan 02 und dem UI in Plan 04. Ohne diesen Plan wären alle UI-Screens in Plan 04 Mock-Layouts: join-by-code hätte keinen RPC-Wrapper, settings/garden könnte keine Member-Liste laden, und migrateLocalToAccount würde nach dem Sign-Up keinen Garden anlegen. Vier atomare Commits liefern:

1. **authStore** bekommt `activeGardenId` + `setActiveGarden` + Zustand persist `version: 1` migrate für v0-Blobs aus Phase 02.
2. **gardenRepo** (neu, 228 Zeilen) mit 6 Abfragen + 4 typed Error-Klassen (`NotOwnerError`, `GardenHasMembersError`, `CannotTransferToSelfError`, `TargetNotMemberError`) — D-16 Owner-Rights voll gemappt auf Supabase-SQLSTATE. **inviteCodeRepo** (neu, 53 Zeilen) ist reine RPC-Wrapper-Schicht mit Confusable-Char-Normalisierung im Client.
3. **profileRepo** account-mode auf `display_name` geschrumpft (D-01 Pivot); **vereinsregelnRepo.toRow** ist jetzt 3-arg `(rule, userId, gardenId)` und stempelt `created_by_user_id` + `updated_by_user_id` + `garden_id`. Load/save/delete pullen `activeGardenId` via `useAuthStore.getState()` und werfen `no_active_garden` wenn null. **enqueueAiJob** parallel gefixt (Plan-02-Deferred).
4. **migrateLocalToAccount** 8-Step-Flow: signUp → ensureDefaultGardenForUser → read storage → profile upsert (display_name only) → gardens.update (plz/klimazone/archetype) → vereinsregeln upsert (3-arg toRow) → setAccountMode + setActiveGarden → storage.delete. Atomic-Tail-Invariant über 5 neue Tests abgesichert.

## migrateLocalToAccount 8-Step Flow (post-Plan-03)

```
1. supabase.auth.signUp({email, password}) → newUserId
2. ensureDefaultGardenForUser() → gardenId                     ← NEU (Phase 2.5)
3. storage.get('profile') + storage.get('vereinsregeln')       ← read-only, storage untouched
4. profiles.upsert({id: newUserId, display_name: emailPrefix}) ← D-01: ONLY display_name
5. gardens.update({plz, klimazone, archetype, updated_by_user_id}).eq('id', gardenId)  ← NEU (Phase 2.5)
6. vereinsregeln.upsert(toRow(r, newUserId, gardenId))         ← 3-arg toRow; re-stamp + garden-scope
7. state.setAccountMode(newUserId) + state.setActiveGarden(gardenId)  ← setActiveGarden NEU
8. storage.delete('profile') + storage.delete('vereinsregeln') ← atomic tail (MUST be last)
```

Error-Codes (partial-failure-states mit local storage intact):

| Step | Error Code | Downstream blocked |
|---|---|---|
| 1 | supabase auth error rethrown | ✓ — keine RPC, kein upsert, kein auth flip |
| 2 | `migration_partial_garden_seed: <rpc msg>` | ✓ — profiles/gardens/vereinsregeln unberührt |
| 4 | `migration_partial_profile: <supabase msg>` | ✓ — gardens/vereinsregeln unberührt |
| 5 | `migration_partial_garden_metadata: <supabase msg>` | ✓ — vereinsregeln unberührt |
| 6 | `migration_partial_vereinsregeln: <supabase msg>` | ✓ — auth-flip passiert nicht, storage intact |

**MigrateResult shape:**
```typescript
{ userId: string; gardenId: string; transferred: { profile: boolean; vereinsregeln: number } }
```
`gardenId` ist NEU in Phase 2.5 und wird von Plan 04 AuthChoiceCard für Post-Migration-Navigation konsumiert.
