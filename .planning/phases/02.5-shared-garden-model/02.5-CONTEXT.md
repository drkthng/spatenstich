# Phase 2.5: Shared Garden Model - Context

**Gathered:** 2026-04-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Daten-Modell-Pivot von single-user (`user_id = auth.uid()`) auf Shared Garden: neue Tabellen `gardens` + `garden_members`, RLS auf Member-Check umgestellt, 6-stelliger Invite-Code-Flow, damit ein zweiter Account (Frau) Dirks Garten beitreten kann. Alle Phase-2-Daten (Profile-Scope, Vereinsregeln, AI-Jobs/Results) werden einem Default-Garten zugeordnet. LWW über `updated_at` + `updated_by_user_id`; UI zeigt "zuletzt bearbeitet von <Name>" inline.

Kein Plan-Editor, keine Foto-Capture, kein Kalender — das ist Phase 4+. Phase 2.5 macht Phase 3 (Offline/Sync mit 2-User-LWW) überhaupt erst möglich.

Max 2 Member pro Garten (hart via DB CHECK). Lokal-Modus bleibt single-user — Shared Garden existiert nur im Account-Modus.

</domain>

<decisions>
## Implementation Decisions

### Garten-Modell & Profil-Scope

- **D-01:** PLZ + Klimazone + Archetyp wandern komplett auf die neue `gardens`-Tabelle. `profiles` behält nur `id` + `display_name` (neu) + `created_at`/`updated_at`. Semantik: ein Kleingarten hat genau eine Lage/Klimazone/Charakter — nicht pro Person.
- **D-02:** Die Phase-2-Tabellen `vereinsregeln`, `ai_jobs`, `ai_results` bekommen einen `garden_id uuid not null references gardens(id) on delete cascade` — RLS wird von `auth.uid() = user_id` auf `auth.uid() in (select user_id from garden_members where garden_id = <row>.garden_id)` umgestellt.
- **D-03:** `gardens`-Tabelle minimal: `id uuid pk`, `name text not null`, `created_at timestamptz`, `updated_at timestamptz`, `created_by_user_id uuid references auth.users(id)` (Owner-Marker). Keine Parzellen-Metadaten (Maße/Form) — das kommt in Phase 4.
- **D-04:** `garden_members`-Tabelle: `garden_id uuid fk`, `user_id uuid fk`, `role text check (role in ('owner','member'))`, `joined_at timestamptz`. Composite PK `(garden_id, user_id)`. DB CHECK constraint oder Trigger, der Member-Count auf 2 limitiert.
- **D-05:** Neue Spalte `display_name text` in `profiles`, User-gesetzt beim Onboarding (oder migriert aus E-Mail-Prefix als Default). Wird für "zuletzt bearbeitet von"-Label genutzt.
- **D-06 (Claude's Discretion):** FK-Struktur auf geteilten Tabellen — `garden_id` (für RLS-Scope) + `updated_by_user_id` (für SC4-Audit, nullable für Legacy-Zeilen). Bestehendes `user_id` auf `vereinsregeln`/`ai_jobs`/`ai_results` kann als `created_by_user_id` umbenannt bleiben oder droppen — entscheidet Researcher je nach Migration-Komplexität.

### Invite-Code-Flow

- **D-07:** Nur Owner (entsprechend `garden_members.role = 'owner'`) darf Invite-Code erzeugen. Member sieht in Settings nur die Member-Liste, keinen Einlade-Button.
- **D-08:** 6-stelliger alphanumerischer Code aus ambiguitätsfreiem Alphabet (kein `0/O`, kein `1/I/l`). Am Telefon diktierbar. UI: großer Code-Display + Copy-Button + Share-Sheet (iOS/Android native).
- **D-09:** 24h gültig, Ein-Use (DB-Spalte `consumed_at timestamptz null`). Nach Beitritt oder Ablauf nicht wiederverwendbar.
- **D-10:** UI-Flow: **Owner-Seite** Settings → "Mein Garten" → Button "Person einladen" → erzeugt Code, zeigt + Copy/Share. **Joiner-Seite** "Code eingeben"-Screen erreichbar (a) im Onboarding als dritte Option neben "Account erstellen" und "Lokal nutzen", (b) nachträglich in Settings für User die schon eingeloggt sind.
- **D-11:** Revokation implizit: "Neuen Code erzeugen" invalidiert den alten (nur ein aktiver Code pro Garten gleichzeitig). Kein separater "Widerrufen"-Button im MVP.

### Migration Phase 2 → 2.5

- **D-12:** Auto-Migration für existing Account-User: beim ersten Login nach dem Phase-2.5-Deploy wird ein Default-Garten `{name: 'Mein Garten', created_by_user_id: <uid>}` angelegt, der User als Owner in `garden_members` eingetragen, und alle `profiles`/`vereinsregeln`/`ai_jobs`/`ai_results`-Zeilen dieses Users werden mit der neuen `garden_id` versehen. Implementiert als DB-Trigger + App-Boot-Logic (Defense-in-Depth). Null Friction für Dirk.
- **D-13:** Lokal-Modus bleibt konzeptionell single-user — `gardens`/`garden_members` existieren nur in Supabase, nicht im StorageAdapter. `migrateLocalToAccount.ts` wird erweitert: am Ende (nach signUp + profile/vereinsregeln-Upsert) wird zusätzlich Default-Garten erzeugt und alle Rows mit `garden_id` gestempelt.
- **D-14:** Frau's eigene Daten (falls sie vor Beitritt eigenen Account oder Lokal-Modus hatte): bleiben unangetastet in ihrem eigenen Space (entsprechender "alter" Default-Garten). Beim Beitritt via Invite-Code wird ein weiterer `garden_members`-Eintrag für Dirks Garten angelegt. UI zeigt nach Beitritt nur Dirks Garten (aktiver Garten via `authStore.activeGardenId`). Merge zwischen Gärten ist Out-of-Scope für MVP.
- **D-15:** RLS-Policy-Switch atomic: in EINER Migration DROP alte `*_own` Policies + CREATE neue Member-Check-Policies + Default-Gärten für bestehende User + `garden_members`-Seed-Rows. Postgres-Transaktion → kein Zeitfenster mit kaputten Policies. Migration-Reihenfolge: gardens+garden_members anlegen → Default-Gärten seeden → `garden_id`-Spalten ALTER TABLE ADD auf Ziel-Tabellen → Bestandsdaten UPDATE SET garden_id → ALTER TABLE SET NOT NULL → DROP alte Policy → CREATE neue Policy.

### Member-Rollen, Limits & LWW-Anzeige

- **D-16:** Rollen: **Owner** darf Invite-Code erzeugen, Member entfernen, Garten löschen, eigenen Owner-Status abgeben. **Member** darf alle Content-Rows bearbeiten (vereinsregeln, später Plan/Inventar/Kalender) + sich selbst aus dem Garten entfernen. Content-Edit-Rechte sind beiden gleich — kein Read-only.
- **D-17:** Max 2 Member pro Garten hart codiert via DB CHECK (oder BEFORE INSERT trigger, je nach Supabase-Kompatibilität des einfacheren Wegs). Lift-Path: einfacher ALTER TABLE + CHECK-Update wenn Post-MVP >2 erlaubt werden soll.
- **D-18:** LWW-Tracking: `updated_at timestamptz` + `updated_by_user_id uuid references auth.users(id)` auf allen Tabellen mit `garden_id` (`gardens`, `vereinsregeln`, `ai_jobs`, `ai_results`). `updated_at` wird per Trigger (`tg_set_updated_at` existiert schon aus Phase 1) gesetzt; `updated_by_user_id` client-seitig bei jedem Upsert mitgesendet.
- **D-19:** UI-Anzeige "zuletzt bearbeitet von <Name>": inline pro Row, kleingedruckt als Metadata (z.B. Vereinsregeln-Liste unter jedem Titel: "zuletzt bearbeitet von Maria, vor 2 h"). Name-Source = `profiles.display_name` des User mit `id = updated_by_user_id`. Muster überträgt sich ab Phase 5 (Plan-Editor) + Phase 7 (Kalender) auf weitere Rows.

### Claude's Discretion

- Default-Garten-Name-Format beim Auto-Trigger: `'Mein Garten'` vs `'Garten von <display_name>'` — entscheide basierend auf i18n-Überlegungen.
- Exact Alphabet für Invite-Code (base32-Variante ohne confusables vs. custom Set).
- QR-Code-Generierung zusätzlich zu Text-Code — nice-to-have für Phase 2.5, aber nicht blockierend.
- Timestamp-Format in "zuletzt bearbeitet von"-Label ("vor 2 h" vs "heute 14:22" vs vollständig) — entscheide basierend auf NativeWind-Pattern in Phase 2.
- Cascade-Verhalten bei Owner-Verlässt-Garten (Remove mit aktivem Member): Ownership transferiert automatisch auf verbleibenden Member, oder Warnung "Garten muss gelöscht werden, wenn du als letzter Owner gehst".
- Exact DB-Mechanismus für 2-Member-Limit: CHECK constraint mit subquery ist in Postgres nicht direkt möglich — nutze BEFORE INSERT trigger oder partial unique index (alternative Pattern-Recherche beim Planner).

</decisions>

## Existing Code Insights

### Reusable Assets
- `app/src/lib/supabase.ts`: Supabase-Client mit persistSession+autoRefreshToken — für auth-Aktionen beim Beitritts-Flow direkt nutzbar
- `app/src/lib/migrateLocalToAccount.ts`: atomic-tail-Pattern (storage.delete strikt NACH allen Supabase-Upserts) ist der Blueprint für den erweiterten Lokal→Account+Garten-Migrationsflow
- `app/src/lib/vereinsregelnRepo.ts`: `toRow`/`fromRow`-Trennung (camelCase-Domain vs snake_case-DB) ist das Pattern, das für `gardenRepo.ts` und alle neuen gardens-gescopeden Repos zu übernehmen ist
- `app/src/stores/authStore.ts`: Zustand + persist-Middleware → einfach erweiterbar um `activeGardenId`
- `public.tg_set_updated_at()` Trigger: vorhanden aus Migration 001 — auf alle neuen Tabellen mit `updated_at` direkt anwendbar
- `packages/shared/src/constants/klimazonen.ts`: PLZ→Klimazone Lookup — Logic wandert vom User-Level zum Garden-Level, aber Lookup-Funktion bleibt

### Established Patterns
- **Mode-aware Repo-Pattern** (profileRepo/vereinsregelnRepo): `if (mode === 'account') supabase else storage` — wird erweitert durch Garten-Scope-Check: `if (account && activeGardenId)` vs `lokal`
- **Re-Stamping bei Migration** (migrateLocalToAccount, T-2-04-02): FK-IDs werden beim Transfer umgeschrieben — identisches Pattern für `garden_id`-Stempel auf migrated rows
- **RLS-Test-Pattern** (Phase 1): SET LOCAL ROLE authenticated in Supabase-Queries, sonst läuft alles als postgres-superuser → MUSS für neue Member-Check-RLS angewandt werden
- **Storage-Adapter als JSON-Blob** (Pitfall 6): Lokal-Daten liegen als ein JSON-Blob pro Key (nicht flache Keys) — bleibt so, da Lokal-Modus kein Garten-Konzept kriegt

### Integration Points
- `app/app/_layout.tsx`: Root-Layout mit Session/Local-UUID-Check — erweitern um `activeGardenId`-Resolution (Auto-Default-Garten beim ersten Login nach Update)
- `app/app/(app)/settings.tsx`: existing Settings-Screen — neue Sektion "Mein Garten" mit Member-Liste + "Person einladen"-Button
- `app/app/(auth)/`: existing Auth-Screens — erweitern um "Einem Garten beitreten"-Option mit Code-Eingabe-Screen
- `supabase/functions/`: falls Server-side Invite-Code-Generierung benötigt → neue Edge Function `create-invite` analog zu existing `extract-vereinsregeln`. Alternative: Client-side RPC via Supabase `rpc()` mit SECURITY DEFINER stored procedure (einfacher, kein Deno-Deploy). Researcher-Entscheidung.

<specifics>
## Specific Ideas

- "Das Onboarding darf nicht auseinanderfallen" — sprich: Dirks existing User-Experience (Phase 2: free navigation, keine Pflichtfelder nach Auth) bleibt unverändert. Phase 2.5 fügt nur die neue Option "Einem Garten beitreten" hinzu, ohne den bestehenden Weg "Account erstellen" + "Lokal nutzen" zu stören.
- "Code am Telefon diktierbar" — User kann Code mündlich weitergeben. Deswegen Alphabet ohne 0/O/1/I/l.
- "Zuletzt bearbeitet von <Name>" ist das sichtbare Signal für Shared-Garden-Awareness — soll nicht versteckt in Menüs oder nur bei Konflikt sichtbar sein, sondern immer inline als Metadata.
- Settings → "Mein Garten" als neuer Screen folgt dem existing Settings-Pattern aus Phase 2-04 (kein Modal, inline confirmation expansion, UI-SPEC line 234).

</specifics>

<deferred>
## Deferred Ideas

- **Multi-Garten-Switching** (ein User ist Member in mehreren Gärten und wechselt zwischen aktiven): Out-of-Scope für MVP; im Datenmodell bereits vorbereitet (`garden_members` viele-zu-viele), aber UI/Store behandelt nur einen aktiven Garten. Post-MVP Phase.
- **Member-spezifische Anzeigenamen** (`display_name_in_garden` pro `garden_members`-Row statt global auf `profiles`): Post-MVP, falls sich ergibt dass User pro Garten anders heißen wollen.
- **Audit-History-Log** (separate Tabelle mit allen Änderungen der letzten N Edits): Post-MVP. Inline "zuletzt bearbeitet von" reicht für MVP.
- **>2 Member pro Garten**: explizit Out-of-Scope laut PROJECT.md. Lift-Path via DB-Constraint-Update später, keine Schema-Migration nötig.
- **Daten-Merge bei Beitritt** (Frau hat schon eigene Vereinsregeln → werden in Dirks Garten mergen): Konfliktgeladen (welche PLZ? welcher Archetyp? dedupen von Vereinsregeln?) — Out-of-Scope.
- **Explizite Invite-Code-Revokation-UI** (Button "Widerrufen" neben aktivem Code): wird implizit über "Neuen Code erzeugen" gelöst; expliziter Button erst bei Bedarf.
- **Rate-Limiting für Invite-Code-Generierung** (z.B. max 10 Codes/Tag pro Owner): Realistisch im 2-User-MVP nicht nötig; Supabase hat generelle Auth-Rate-Limits. Bei Missbrauchs-Anzeichen nachziehen.

</deferred>
